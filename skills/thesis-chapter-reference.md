## 工作流程

### 步骤 1: 确定要修改的章节

首先明确要修改的论文章节（如 3.2.1、3.4.2 等）。

### 步骤 2: 检查是否存在参考文档

使用 Glob 工具搜索 `/home/haojie/workspace/lcmhalmcp/doc/` 目录下是否有相关文档：

```bash
# 搜索文档（根据关键词）
glob: /home/haojie/workspace/lcmhalmcp/doc/*mmio*
glob: /home/haojie/workspace/lcmhalmcp/doc/*static*
glob: /home/haojie/workspace/lcmhalmcp/doc/*dynamic*
glob: /home/haojie/workspace/lcmhalmcp/doc/*arch*
```

### 步骤 3: 读取参考文档内容

如果找到相关文档，使用 Read 工具读取文档内容：

```bash
read: /home/haojie/workspace/lcmhalmcp/doc/<document-name>.md
```

重点关注：
- **实现细节**：具体的实现技术（如 CodeQL 查询规则、数据流追踪方式）
- **代码示例**：实际的代码片段和证据路径
- **设计要点**：关键设计决策和技术选型理由
- **仓库证据**：具体文件路径和行号，用于验证

### 步骤 4: 基于参考文档进行修改

根据参考文档的内容，对论文进行修改：

1. **技术准确性**：确保描述的技术细节与实际实现一致
2. **代码示例**：参考文档中的代码示例，确保示例真实可信
3. **术语一致性**：使用与代码仓库一致的术语和命名
4. **证据支持**：关键论断应有参考文档中的证据支持

### 步骤 5: 验证修改后的一致性

修改完成后，验证修改内容是否与参考文档一致：

- 技术描述是否准确
- 代码示例是否来自真实代码
- 设计决策是否与实际实现相符
- 术语使用是否一致

## 重要注意事项

1. **优先级**：参考文档的内容优先于论文中现有描述
2. **文档时效性**：注意参考文档的日期，优先使用较新的文档
3. **实现细节**：论文中涉及实现技术时（如 CodeQL 查询、数据流分析），必须有参考文档支持
4. **代码示例**：添加代码示例时，优先从参考文档中选取真实示例

### 论文写作规范（新增）

5. **避免项目文档描述**：不要在论文中写"查询文件位于 queries/collectors/mmio/ 目录"、"典型查询会 import libraries.MMIOAnalyzer"这类项目实现细节描述，论文应专注于技术方法和设计思路

6. **避免"识别特征是"描述**：不要写"识别特征是出现类似..."这种描述，应改为"这类代码在代码中呈现为..."、"在代码中表现为..."等更自然的描述方式

7. **技术描述的论文化**：
   - 描述技术实现时，说明其设计思路和工作原理，而非项目文件结构
   - 使用"查询规则设计思想为..."、"查询通过数据流追踪..."等技术性表述
   - 避免提及具体的文件路径、目录结构、import 语句等实现细节
   - 如需引用具体代码示例，从参考文档中选取并说明其技术意义

8. **代码示例的目的性**：添加代码示例时，必须说明该示例展示了什么技术点，示例应服务于论文论述，而非单纯展示代码
