\section{驱动函数接口静态分析与识别模块实现}

\subsection{CodeQL 查询实现}

本节详细阐述分析与识别模块的具体实现技术。模块采用 CodeQL 静态分析引擎作为底层支撑，CodeQL 通过构建包含语法树、控制流图、数据流图的综合数据库，支持对 C/C++ 代码的语义级查询分析。

\subsubsection{MMIO 操作定位查询实现}

针对三种 MMIO 定位模式，本研究设计了相应的查询规则。查询从 CodeQL 数据库中提取所有符合 MMIO 结构体特征的变量及其初始化表达式。查询的核心设计思想是将"寄存器字段访问"追溯到一个可被 CodeQL 稳定描述的基对象表达式，如代码~\ref{lst:mmio-collect}所示。

\begin{lstlisting}[style=CodeQL,language=CodeQL, caption={MMIO表达式收集查询示例},label=lst:mmio-collect]
import cpp
import libraries.MMIOAnalyzer

from
    Function func,
    InterestingMMIOExpr expr
where
    isInterestingMMIOFunction(func) and
    expr.getEnclosingFunction() = func
select expr.toString(),
       expr.getEnclosingFunction().getName(),
       expr.getFile().getAbsolutePath().toString(),
       expr.getLocation().getStartLine(),
       expr.checkExprType(),
       expr.checkEnclosingType()
\end{lstlisting}

该查询通过 \texttt{import libraries.MMIOAnalyzer} 引入 MMIO 分析库，从 CodeQL 数据库中提取所有符合条件的 MMIO 表达式。查询的输出包括表达式字符串、所属函数名、文件路径、行号、表达式类型和封闭类型等信息。

对于"间接初始化"和"常量指针参数"模式，查询采用数据流追踪策略，通过追踪赋值语句和函数参数来识别 MMIO 实例。常量基址的追踪利用 CodeQL 提供的数据流和污点追踪能力，将"常量大值表达式（外设基址）"追踪到后续的变量访问/解引用位置，如代码~\ref{lst:taint-tracking}所示。

\begin{lstlisting}[style=CodeQL,language=CodeQL, caption={常量基址数据流追踪查询示例},label=lst:taint-tracking]
 predicate isMMIOConstantExpr(ConstantExpr src_mmio) {
    exists( DataFlow::Node source, DataFlow::Node sink, Expr varAccess,
             PointerDereferenceExpr deref |
        source.asExpr() = src_mmio and
        isConstantBigValueExpr(src_mmio) and
        sink.asExpr() = varAccess and
        deref.getOperand() = varAccess and
        TaintFlow::flow(source, sink) and
        not sink.asExpr().getFile() instanceof DriverFile
    )
}

predicate isMMIOTracedExpr(Expr e) {
    exists( DataFlow::Node source, DataFlow::Node sink, ConstantExpr src_mmio |
        source.asExpr() = src_mmio and
        isMMIOConstantExpr(src_mmio) and
        sink.asExpr() = e and
        TaintFlow::flow(source, sink) and
        not sink.asExpr().getFile() instanceof DriverFile
    )
}
\end{lstlisting}

该查询定义了两个谓词：\texttt{isMMIOConstantExpr} 识别常量大值表达式（外设基址），\texttt{isMMIOTracedExpr} 通过污点追踪将源点追踪到汇点。查询的逻辑可形式化表示为：当存在一个常量大值表达式作为数据流源点，经过数据流传播到达某个变量访问或解引用操作时，该操作即为 MMIO 相关的表达式。

\begin{lstlisting}[style=CStyle,language=C,caption=常量基址在代码中的传播示例,label=lst:mmio-propagation]
// 示例：常量基址在代码中的传播
#define USART1_BASE 0x40011000UL
USART_TypeDef *USART1 = (USART_TypeDef *)USART1_BASE;  // 基址强转
huart->Instance = USART1;                              // 赋值给Instance成员
huart->Instance->DR = data;                            // 最终的字段访问点
\end{lstlisting}

查询通过污点追踪，将上述代码中的 \texttt{USART1\_BASE}（源点）追踪到最终的 \texttt{huart->Instance->DR}（汇点），从而识别出整个 MMIO 使用链路，如代码~\ref{lst:mmio-propagation}所示。

对于"函数第 0 个参数是指针类型，实参是常量大值表达式"的情况，查询通过函数调用关系和参数类型匹配，定位"把常量基址传给接受结构体指针的驱动函数"的调用点，如代码~\ref{lst:param-type-match}所示。

\begin{lstlisting}[style=CodeQL,language=CodeQL, caption={函数参数类型匹配查询示例},label=lst:param-type-match]
  predicate isMMIOStructure_MK(Struct s){
     exists( FunctionCall call, Expr e, TypedefType typeDef |
        call.getTarget().getParameter(0).getType() instanceof PointerType and
        (
            (call.getTarget().getParameter(0).getType().(PointerType).getBaseType() = typeDef and
             typeDef.getBaseType() = s) or
            call.getTarget().getParameter(0).getType().(PointerType).getBaseType() = s
        ) and
        call.getArgument(0) = e and
        isConstantBigValueExpr(e)
     )
 }
\end{lstlisting}

该查询通过参数类型匹配（\texttt{PointerType}）和实参的常量大值表达式检查，识别出 MMIO 结构体类型的函数调用点。查询的逻辑可表示为：当函数调用的第 0 个参数类型为结构体指针，且该参数对应的实参是经过数据流追踪的 MMIO 常量表达式时，该调用点即为候选的 MMIO 使用点。

\begin{lstlisting}[style=CStyle,language=C,caption=常量指针参数传递示例,label=lst:pointer-param-pass]
// 示例：常量指针参数传递
void uart_send_data(USART_TypeDef *uart, uint8_t *data, uint16_t len);

// 调用时传递MMIO实例
uart_send_data(USART1, buffer, len);     // 直接传强转实例
uart_send_data(huart->Instance, buffer, len);  // 传Instance成员
\end{lstlisting}

查询通过参数类型匹配（\texttt{USART\_TypeDef *}）和实参的数据流追踪，识别出上述调用点中的 MMIO 参数传递关系，如代码~\ref{lst:pointer-param-pass}所示。

\subsubsection{驱动函数定位查询实现}

驱动函数定位模块的核心任务是根据前述定位准则，从固件源代码中识别出所有驱动函数，并构建驱动函数调用图。模块采用 CodeQL 静态分析技术，通过定义驱动函数的判定规则来实现自动化识别。

模块的核心判定规则是：\textbf{包含 \texttt{DriverFieldAccess} 且不包含 \texttt{MMIOFieldAccess} 的函数被定义为驱动函数}。\texttt{DriverFieldAccess} 指的是通过句柄或间接方式访问 MMIO 寄存器块的字段操作，而 \texttt{MMIOFieldAccess} 指的是直接通过 MMIO 结构体指针访问寄存器的字段操作。这一定义确保了识别出的函数是真正需要替换的驱动层函数，而非底层的硬件抽象层函数。判定规则的 CodeQL 实现如代码~\ref{lst:driver-function-rule}所示。

\begin{lstlisting}[style=CodeQL,language=CodeQL, caption={驱动函数判定规则},label=lst:driver-function-rule]
class DriverFunction extends Function
{
    DriverFunction() {
        exists(DriverFieldAccess fa | fa.getEnclosingFunction() = this) and
        not exists(MMIOFieldAccess fa | fa.getEnclosingFunction() = this)
    }
}
\end{lstlisting}

该查询定义了 \texttt{DriverFunction} 类，该类继承自 \texttt{Function}，通过检查函数内部是否包含 \texttt{DriverFieldAccess} 且不包含 \texttt{MMIOFieldAccess} 来判定函数是否为驱动函数。

在识别出驱动函数列表后，模块进一步分析驱动函数之间的调用关系，构建函数调用图。调用图以驱动函数为节点，函数调用关系为边，记录调用方、被调用方、调用位置等信息。调用图记录了函数间的调用依赖关系，有助于理解函数的调用上下文和语义关联。

\begin{lstlisting}[style=CodeQL,language=CodeQL, caption={驱动函数调用关系查询},label=lst:driver-call-query]
from DriverToCall call
select call.toString2(),
       call.getTargetFunction().toString(),
       call.getEnclosingFunction().toString(),
       call.isDirectCall()
\end{lstlisting}

该查询从 \texttt{DriverToCall} 关系中提取驱动函数之间的调用关系，输出调用边的字符串表示、目标函数名、封装函数名以及调用是否为直接调用等信息，如代码~\ref{lst:driver-call-query}所示。

\subsection{数据模型实现}

为便于后续模块使用，静态分析结果被组织为统一的数据模型。数据模型采用 Python 数据类（dataclass）定义，支持序列化和反序列化，如代码~\ref{lst:data-model}所示。

\begin{lstlisting}[style=Python,language=Python,caption=数据模型定义示例,label=lst:data-model]
# 示例：数据模型定义
@dataclass
class MmioAccessInfo:
    struct_type: str      # 结构体类型，如"USART_TypeDef"
    field_name: str       # 字段名，如"DR"
    access_type: str      # 访问类型：READ/WRITE
    line_number: int      # 行号
    function_name: str    # 所在函数名
    context_code: str     # 上下文代码

@dataclass
class FunctionInfo:
    name: str             # 函数名
    file_path: str        # 文件路径
    line_number: int      # 行号
    signature: str         # 函数签名
    mmio_accesses: List[MmioAccessInfo]  # MMIO访问列表
\end{lstlisting}

数据模型包括函数信息模型（包含函数名、文件路径、行号、签名、返回类型、参数列表、分类、源码、调用关系等信息）、MMIO 访问点模型（包含结构体类型、字段名、访问类型、行号、所在函数、上下文代码等）和项目分析结果模型（包含项目路径、CodeQL 数据库路径、所有函数、MMIO 访问点、函数调用图、驱动函数列表、分析时间戳等）。这些数据模型通过 JSON 序列化存储在缓存层，供后续模块查询和使用。

\subsubsection{查询工具函数实现}

各模块的数据通过一组工具函数对外暴露，供分析智能体和修复智能体在 LangGraph 工作流中调用。工具函数基于 LangChain 的 \texttt{@tool} 装饰器实现，将底层的数据查询封装为 LLM 可直接调用的工具接口。以下列举系统中的主要工具函数。

\textbf{GetFunctionInfo} 工具接收函数名作为输入，从静态分析结果中检索对应的 FunctionInfo 对象，返回函数的完整源码、签名、参数列表、返回类型、所在文件和行号等信息。当查询的函数名不存在时，工具返回空结果并提示可能的函数名变体（如去掉前缀/后缀后的匹配），供 LLM 纠正查询参数。

\textbf{GetMMIOFunctionInfo} 工具接收函数名作为输入，返回静态分析阶段提取的该函数所有 MMIO 访问点信息，包括访问的结构体类型、字段名、访问类型（读/写）和上下文代码，供分析智能体在分类阶段判断函数与硬件交互的具体方式。

\textbf{GetFunctionCallStack} 工具接收函数名作为输入，从静态分析构建的函数调用图中提取该函数的调用者列表和被调用者列表，返回调用关系的双向视图，供分析智能体理解函数在驱动架构中的定位。

除以上基于静态分析结果的查询工具外，系统还提供了基于仿真运行结果的查询工具。\textbf{GetMMIOFunctionEmulateInfo} 返回 MMIO 函数在仿真运行中的调用日志（包括调用次数、参数值和返回值），\textbf{GetFunctionCallsEmulateInfo} 返回仿真运行中的函数调用栈信息。这两个工具的数据来源为仿真运行阶段采集的运行时日志，而非静态分析结果，主要用于仿真执行智能体在故障诊断时定位问题函数的调用上下文。

此外，系统还提供 GetStructOrEnumInfo（查询数据结构定义）、GetDriverInfo（获取驱动模块上下文）等辅助工具，供智能体按需调用。各工具函数的返回结果均经过格式化处理，以结构化的文本形式呈现给 LLM，避免原始 JSON 数据对模型推理的干扰。基于静态分析的工具函数内部实现了缓存机制，对同一函数的重复查询直接返回缓存结果，减少对 CodeQL 数据库的重复访问。

\subsection{分析流程实现}

静态分析模块的分析流程包括四个主要步骤：CodeQL 数据库构建、查询执行、结果聚合和模型存储。

\textbf{步骤1：数据库构建}。调用 \texttt{codeql database create} 命令构建 CodeQL 数据库，该过程包括源码解析、AST 生成、控制流图和数据流图的构建。数据库构建完成后，可以通过 CodeQL 查询语言对代码进行语义级分析。

\textbf{步骤2：查询执行}。依次执行前述设计的查询规则，包括 MMIO 表达式收集查询、常量基址数据流追踪查询、驱动函数判定查询和调用关系查询。每个查询返回一组结果，这些结果被转换为 Python 对象供后续处理。

\textbf{步骤3：结果聚合}。将各个查询的结果进行聚合和关联。例如，将 MMIO 表达式与函数信息关联，构建完整的函数-MMIO 访问映射；将驱动函数与调用关系关联，构建函数调用图。

\textbf{步骤4：模型存储}。将聚合后的结果存储到统一的数据模型中，并通过 JSON 序列化保存到文件。存储的数据模型包括项目分析结果、函数信息列表、MMIO 访问点列表和调用关系图。这些数据通过文件系统缓存，支持后续模块的快速查询和断点续跑。

\section{驱动函数分析、替换与编译模块实现}

\subsection{智能体工作流实现}

本节阐述基于 LangGraph 框架的智能体工作流实现。LangGraph 是一个用于构建有状态、多智能体应用的框架，它通过图（Graph）来表示应用的状态转换和逻辑流，特别适合本系统这种包含多轮迭代和条件分支的复杂场景。

\subsubsection{驱动函数分析与替换代码生成模块实现}

驱动函数分析与替换代码生成模块的设计逻辑见第~\ref{sec:analyzer-workflow}~节，本节阐述其基于 LangGraph 框架的具体实现。该模块的核心功能是以单个驱动函数为粒度，通过分析智能体（Analyzer Agent，下文简称分析智能体）依次完成信息获取、函数分类和替换代码生成三个阶段，实现从原始驱动函数到仿真兼容实现的自动化转换。

分析智能体基于 LangGraph 的 \texttt{StateGraph} 类构建，采用图式状态机架构。智能体的核心状态包括当前处理的函数信息、函数分类结果、替换代码、验证结果和错误信息。智能体的转换逻辑由条件函数决定，根据验证结果决定是否需要重新生成替换代码或进入下一个处理阶段。在实现上，智能体通过 \texttt{@tool} 装饰器将 GetFunctionInfo、GetFunctionCallStack、GetMMIOFunctionEmulateInfo 等工具函数注册到 LangGraph 框架中，通过 \texttt{compile()} 方法编译为可执行的 \texttt{CompiledGraph} 对象，由执行器调度节点的执行顺序并管理状态传递。

\subsubsection{替换代码修正与编译集成模块实现}
\label{sec:builder-fixer-impl}

替换代码修正与编译集成模块的设计逻辑见第~\ref{sec:builder-workflow}~节，本节阐述其具体实现。该模块负责对上一模块输出的候选替换代码进行二阶段检验与修复，由编译修复智能体（BuilderFixer Agent，下文简称编译修复智能体）统一完成代码集成、编译构建和错误修复，目标是在真实构建环境下得到可编译、可链接的替换定义。

\textbf{代码集成机制}：系统采用"原位替换"策略将替换代码写入工程。对于每个需要替换的驱动函数，系统在目标源文件中保留原始函数定义的快照（用于回滚），然后将替换实现写入函数体。同时，系统在工程中添加弱符号桩函数文件，为仿真注入接口（\texttt{HAL\_BE\_*} 系列函数）提供链接占位，确保在非仿真环境中不会因未定义符号而导致链接失败。这种设计使得同一个工程既可以在原始硬件环境中编译运行，也可以在仿真环境中使用替换代码运行，无需维护两套代码副本。

\textbf{编译构建与错误修复流程}：编译修复智能体将替换代码应用到工程后执行编译构建，若编译失败则自动进入错误解析与修复循环。替换代码应用通过 ReplaceFuncs 工具实现，准确定位函数在源文件中的位置、保留注释和格式、生成回滚快照；工程构建通过 BuildProject 工具执行 CMake 构建，配置 ARM 工具链环境并捕获编译输出。内置的编译错误解析器支持 GCC、Clang、ARMCC 等多种编译器，能够将诊断信息转换为包含错误类型、位置和消息的结构化错误对象。编译失败时，编译修复智能体自动解析编译器输出的错误信息，定位到具体的替换函数和错误原因，然后生成针对性的修复方案并重新构建，形成"编译 $\to$ 错误解析 $\to$ 修复 $\to$ 重新编译"的闭环。修复策略的选择由 LLM 驱动，LLM 根据错误信息和函数上下文智能地选择补声明、改类型与转换、补 \texttt{\#include}、删改非法成员访问等策略。编译修复智能体采用"锁定首个错误"策略，优先修复编译器报告的第一个错误所涉及的函数，避免同时处理多个错误导致修复方案相互冲突。编译错误的典型成因包括：函数未声明或声明不匹配、类型不兼容（如 LLM 将 \texttt{uint32\_t} 参数改写为 \texttt{int}）、缺少头文件包含、替换代码引用了目标芯片外设库中不存在的结构体成员等。每次修复后重新触发编译，直至编译成功或达到最大修复轮次。

\subsubsection{替换代码验证机制实现}

第~\ref{chap:design}~章第~3.2.3~节定义了替换代码的四层验证机制，本节阐述其具体实现。

\textbf{语法正确性检查}（$V_1$）通过 Python 的正则表达式和括号匹配算法实现。验证器对替换代码进行以下检查：括号（花括号、圆括号、方括号）是否匹配；语句是否以分号结尾（排除控制结构行）；是否存在未闭合的字符串字面量；函数体是否非空。对于语法检查未通过的代码，验证器返回具体的语法错误描述（如"第 3 行缺少右花括号"），帮助 LLM 在下一轮生成中定位修正。

\textbf{签名匹配检查}（$V_2$）通过将替换代码的函数签名与静态分析模块提供的原始函数签名进行逐字段比对实现。签名提取器通过正则表达式从替换代码的首行解析函数名、返回类型和参数列表，然后与 FunctionInfo 中记录的签名进行匹配。检查内容包括：函数名完全一致、返回类型一致、参数数量一致、每个参数的类型和名称一致。常见的不匹配情形包括 LLM 遗漏参数、将指针参数写为值参数、将 \texttt{const} 修饰符遗漏等。

\textbf{返回值检查}（$V_3$）通过遍历替换函数中所有执行路径的返回语句实现。验证器首先检查函数的返回类型：对于 \texttt{void} 类型函数，验证不存在带值的 \texttt{return} 语句；对于非 \texttt{void} 类型函数，验证所有执行路径（通过简化的控制流分析识别分支和循环的出口）均包含返回语句，且返回值的类型与声明的返回类型兼容。例如，返回类型为 \texttt{HAL\_StatusTypeDef} 的函数，所有返回值应为该枚举的有效成员（如 \texttt{HAL\_OK}、\texttt{HAL\_ERROR}、\texttt{HAL\_TIMEOUT}）。

\textbf{替换规则检查}（$V_4$）根据函数的分类类别执行策略特定的约束检查。各策略的检查规则包括：Skip 策略——替换代码不应包含任何 MMIO 寄存器访问（通过检查是否包含结构体指针的解引用操作 \texttt{->}）；Redirect 策略——替换代码必须包含至少一个 \texttt{HAL\_BE\_*} 系列注入函数调用；Event 策略——替换代码必须保留回调函数的调用；ReturnOK 策略——替换代码的所有返回语句必须返回成功状态值。规则检查器的实现基于正则表达式模式匹配和关键字检索，对未满足策略约束的替换代码生成具体的违规描述。

四层验证通过后，验证器生成通过报告，替换代码进入后续的编译集成流程。若任一层验证失败，验证器将失败原因和修改建议附加到分析智能体的上下文中，触发重新生成。

\subsection{替换策略实现}

各替换策略的设计意图与语义定义见第~\ref{sec:driver-replacement-strategy}~节，本节通过代码示例展示各策略的具体替换效果。

\subsubsection{Skip 策略实现}

Skip 策略适用于两类场景：初始化类函数（INIT）和轮询等待类函数（LOOP）。对于初始化类函数，Skip 策略移除所有硬件寄存器配置操作，仅保留函数签名，使调用路径不被中断，同时通过参数透传确保结构体指针在后续代码中仍然可用。对于轮询等待类函数，Skip 策略移除基于硬件标志位的 while 循环等待逻辑，直接返回成功状态（HAL\_OK），避免仿真环境中因硬件标志永远不置位而导致死循环。代码~\ref{lst:skip-strategy}展示了 Skip 策略在 UART 初始化和轮询函数上的替换效果。

\begin{lstlisting}[style=CStyle,abovecaptionskip=0pt,belowcaptionskip=0pt,caption=Skip 策略替换示例（初始化类和轮询类）,label=lst:skip-strategy]
// 原始代码（初始化类）
void HAL_UART_Enable(UART_HandleTypeDef *huart) {
    huart->Instance->CR1 |= USART_CR1_UE;
}

// 替换代码（Skip 策略）
void HAL_UART_Enable(UART_HandleTypeDef *huart) {
    (void)huart;
}

// 原始代码（轮询类）
HAL_StatusTypeDef HAL_UART_WaitOnFlag(UART_HandleTypeDef *huart, uint32_t Flag, uint32_t Timeout) {
    while (!(__HAL_UART_GET_FLAG(huart, Flag))) {
        if (Timeout != HAL_MAX_DELAY) {
            if ((Timeout--) == 0) return HAL_TIMEOUT;
        }
    }
    return HAL_OK;
}

// 替换代码（Skip 策略）
HAL_StatusTypeDef HAL_UART_WaitOnFlag(UART_HandleTypeDef *huart, uint32_t Flag, uint32_t Timeout) {
    (void)huart;
    (void)Flag;
    (void)Timeout;
    return HAL_OK;
}
\end{lstlisting}

\subsubsection{Redirect 策略实现}

Redirect 策略适用于接收类函数（RECV），其核心思想是将硬件寄存器轮询替换为仿真注入函数调用，同时保留函数中不影响 I/O 的业务逻辑（如状态字段更新、计数器维护等）。对于定长接收场景，使用 HAL\_BE\_In 将仿真输入数据填充到目标缓冲区，覆盖原始的寄存器读取逻辑；对于变长报文接收场景（如以太网驱动），使用 HAL\_BE\_ENET\_ReadFrame 在当前有效 buffer 描述符处注入一帧网络数据。代码~\ref{lst:redirect-uart}展示了 UART 接收函数的替换效果，代码~\ref{lst:redirect-enet}展示了变长报文接收场景的替换效果。

\begin{lstlisting}[style=CStyle,abovecaptionskip=0pt,belowcaptionskip=0pt,caption=Redirect 策略替换示例（UART 接收函数）,label=lst:redirect-uart]
// 原始代码
HAL_StatusTypeDef HAL_UART_Receive(UART_HandleTypeDef *huart, uint8_t *pData,
                                    uint16_t Size, uint32_t Timeout) {
    while (!(huart->Instance->SR & USART_SR_RXNE));
    *pData = huart->Instance->DR;
    return HAL_OK;
}

// 替换代码（Redirect 策略）
HAL_StatusTypeDef HAL_UART_Receive(UART_HandleTypeDef *huart, uint8_t *pData,
                                    uint16_t Size, uint32_t Timeout) {

    HAL_BE_In(pData, Size);
    // Preserve non-I/O logic
    huart->RxXferCount = 0;
    huart->RxXferSize = Size;
    huart->RxState = HAL_UART_STATE_READY;
    return HAL_OK;
}
\end{lstlisting}

\begin{lstlisting}[style=CStyle,abovecaptionskip=0pt,belowcaptionskip=0pt,caption=Redirect 策略替换示例（变长报文接收）,label=lst:redirect-enet]

\subsubsection{Event 策略实现}

Event 策略适用于中断处理和回调函数类（IRQ），其核心思想是绕过硬件中断标志的判断条件，直接触发事件通知和回调函数。在原始代码中，中断处理函数通常通过检查硬件中断标志寄存器（如 ETH\_DMA\_FLAG\_R）来决定是否执行回调，而仿真环境中这些硬件标志永远不会被置位，导致回调逻辑永远无法触发。Event 策略通过移除硬件标志判断，直接调用目标回调函数，使中断处理逻辑能够在仿真环境中正常执行。代码~\ref{lst:event-strategy}展示了 ETH 中断处理函数的替换效果。

\begin{lstlisting}[style=CStyle,abovecaptionskip=0pt,belowcaptionskip=0pt,caption=Event 策略替换示例,label=lst:event-strategy]
// 原始代码
void HAL_ETH_IRQHandler(ETH_HandleTypeDef *heth) {
    if (__HAL_ETH_DMA_GET_FLAG(heth, ETH_DMA_FLAG_R)) {
        HAL_ETH_RxCpltCallback(heth);
    }
    __HAL_ETH_DMA_CLEAR_FLAG(heth, ETH_DMA_FLAG_R);
}

// 替换代码（Event 策略）
void HAL_ETH_IRQHandler(ETH_HandleTypeDef *heth) {
    // Trigger Rx complete callback event directly
    HAL_ETH_RxCpltCallback(heth);
    // No need to check hardware flags or clear registers
}
\end{lstlisting}

\subsubsection{Steering 策略实现}

Steering 策略适用于依赖复杂硬件状态机的场景，其核心思想是通过控制流改写将因硬件依赖而"走不到正确路径"的执行导向到"仿真可达且语义保持"的分支。典型的应用场景是 DMA+RingBuffer 机制：原始代码通过轮询硬件 buffer descriptor 的状态标志位来遍历接收数据，但在仿真环境中这些标志位不会更新，导致循环无法终止。Steering 策略跳过依赖硬件标志的状态扫描循环，在当前 buffer descriptor 处直接调用 HAL\_BE\_ENET\_ReadFrame 注入仿真数据，同时保留 Ring 索引更新和状态字段维护逻辑，确保数据传输和状态更新的语义一致性。代码~\ref{lst:steering-original}和代码~\ref{lst:steering-replaced}展示了 NXP 网卡驱动中 ENET\_GetRxFrame 函数的原始代码和替换后的代码对比。

\begin{lstlisting}[style=CStyle,abovecaptionskip=0pt,belowcaptionskip=0pt,caption=Steering 策略原始代码示例（NXP ENET\_GetRxFrame）,label=lst:steering-original]
// 原始代码：依赖硬件标志扫描 Ring，仿真中将陷入死循环
index = rxBdRing->rxGenIdx;
while (0U == (curBuffDescrip->control & ENET_BUFFDESCRIPTOR_RX_EMPTY_MASK))
{
    if (0U != (curBuffDescrip->control & ENET_BUFFDESCRIPTOR_RX_LAST_MASK)) {
        result = ENET_GetRxFrameErr(curBuffDescrip, &rxFrame->rxFrameError);
        break;
    }
    index = ENET_IncreaseIndex(index, rxBdRing->rxRingLen);
    curBuffDescrip = rxBdRing->rxBdBase + index;
    if (index == rxBdRing->rxGenIdx) break;
}

// 后续：遍历 BD，拷贝数据到 rxFrame
do {
    newBuff = handle->rxBuffAlloc(base, handle->userData, ringId);
    if (newBuff != 0U) {
        rxBuffer = &rxFrame->rxBuffArray[index];
        if (0U != (curBuffDescrip->control & ENET_BUFFDESCRIPTOR_RX_LAST_MASK)) {
            isLastBuff = true;
            rxFrame->totLen = curBuffDescrip->length;
            rxBuffer->length = curBuffDescrip->length - buffLen;
        } else {
            rxBuffer->length = curBuffDescrip->length;
            buffLen += rxBuffer->length;
        }
        // ... buffer 管理、BD 状态更新 ...
    }
} while (!isLastBuff);
\end{lstlisting}

替换后的代码如代码~\ref{lst:steering-replaced}所示，跳过依赖硬件标志的 while 循环，在当前 buffer descriptor 调用 HAL\_BE\_ENET\_ReadFrame 注入数据，并保留 Ring 更新逻辑。

\begin{lstlisting}[style=CStyle,abovecaptionskip=0pt,belowcaptionskip=0pt,caption=Steering 策略替换代码示例（NXP ENET\_GetRxFrame）,label=lst:steering-replaced]
// 替换代码：跳过硬件标志扫描，直接注入数据
curBuffDescrip = rxBdRing->rxBdBase + rxBdRing->rxGenIdx;
index = 0;
do {
    newBuff = (uintptr_t)(uint8_t *)handle->rxBuffAlloc(base, handle->userData, ringId);
    if (newBuff != 0U) {
        rxBuffer = &rxFrame->rxBuffArray[index];
        rxBuffer->buffer = (void *)(uint8_t *)address;

        if (0U != (curBuffDescrip->control & ENET_BUFFDESCRIPTOR_RX_LAST_MASK)) {
            isLastBuff       = true;
            rxFrame->totLen  = curBuffDescrip->length;
            rxBuffer->length = curBuffDescrip->length - buffLen;
        } else {
            rxBuffer->length = curBuffDescrip->length;
            buffLen += rxBuffer->length;
        }
    }

    // Steering 核心：注入仿真数据
    if (rxBuffer->buffer != NULL && rxBuffer->length > 0)
        HAL_BE_ENET_ReadFrame(rxBuffer->buffer, rxBuffer->length);

    // 保留 Ring 更新，防止卡在同一 BD
    index++;
    rxBdRing->rxGenIdx = ENET_IncreaseIndex(rxBdRing->rxGenIdx, rxBdRing->rxRingLen);
    curBuffDescrip     = rxBdRing->rxBdBase + rxBdRing->rxGenIdx;
} while (!isLastBuff);
\end{lstlisting}

\subsubsection{ReturnOK 策略实现}

ReturnOK 策略适用于 RETURNOK 类函数，这类函数的主要功能是执行简单的硬件操作并返回状态值（如 HAL\_OK 或 HAL\_ERROR），如 GPIO 配置、外设复位等。在仿真环境中，这些硬件操作既无实际效果也无必要执行，因此 ReturnOK 策略保留函数签名和必要的参数校验（如空指针检查），直接返回成功状态值，跳过所有硬件相关操作。这样既保证了调用方通过返回值判断执行流程的连贯性，又避免了不必要的硬件依赖。代码~\ref{lst:returnok-strategy}展示了 GPIO 去初始化函数的替换效果。

\begin{lstlisting}[style=CStyle,abovecaptionskip=0pt,belowcaptionskip=0pt,caption=ReturnOK 策略替换示例,label=lst:returnok-strategy]
    if (GPIOx == NULL) return HAL_ERROR;
    GPIOx->MODER = 0x00000000;
    GPIOx->OTYPER = 0x00000000;
    // ... more register operations ...
    return HAL_OK;
}

// 替换代码（ReturnOK 策略）
HAL_StatusTypeDef HAL_GPIO_DeInit(GPIO_TypeDef *GPIOx) {

    return HAL_OK;
}
\end{lstlisting}

\subsection{LLM 提示词工程实现}
\label{sec:prompt-engineering}

大语言模型的分类和生成质量直接影响替换代码的正确性，因此提示词的设计是系统的关键技术实现之一。本节阐述分析智能体和修复智能体的提示词模板结构设计。提示词工程的目标是：（1）为 LLM 提供函数上下文信息，使其能够理解函数语义；（2）通过结构化的分类规则指导 LLM 做出一致的分类决策；（3）通过生成约束限制 LLM 的输出格式，降低后处理复杂度。

\subsubsection{分类提示词模板设计}

分析智能体的提示词模板采用分阶段结构设计，包含角色定义、工具说明、分类策略和输出格式四个部分。

\textbf{角色定义}。提示词以角色设定开场，明确 LLM 扮演嵌入式软件工程师角色，任务是将驱动库函数中的硬件依赖消除，同时保持固件正常运行和 MCU 相关操作（包括操作系统调度和中断触发）。角色定义约束了 LLM 的推理方向，避免其生成与任务无关的输出。

\textbf{工具说明}。提示词中声明了分析智能体可用的五类信息获取工具及其调用时机：GetFunctionInfo（获取函数实现代码，首先调用）、GetMMIOFunctionInfo（检查 MMIO 操作，紧随其后调用）、GetStructOrEnumInfo（查询数据结构定义，按需调用）、GetFunctionCallStack（获取调用关系，按需调用）、GetDriverInfo（获取驱动模块上下文，按需调用）。提示词明确规定了工具调用顺序：GetFunctionInfo $\to$ GetMMIOFunctionInfo $\to$ [GetStructOrEnumInfo | GetFunctionCallStack | GetDriverInfo]（按需选择），确保 LLM 按照固定的信息获取流程执行，避免遗漏关键信息。

\textbf{分类策略}。提示词中嵌入了前述八类函数（RECV、IRQ、INIT、LOOP、RETURNOK、SKIP、CORE、NODRIVER）的识别标准和替换策略。每类函数的定义包含：识别特征（函数名关键词、MMIO 访问模式、控制流结构）、替换策略（Skip、Redirect、Event、Steering、ReturnOK 等）、典型替换示例。分类策略按照优先级排列，提示词中明确要求 LLM 先判断是否属于 RECV 或 IRQ 类（这两类函数的替换策略最复杂，分类错误的影响也最大），再判断其他类别，从而减少高影响函数的误分类率。

\textbf{代码生成约束}。对于需要生成替换代码的函数类别（RECV、IRQ、INIT、LOOP），提示词设置了以下约束：（1）替换代码只能使用原函数中出现的类型和变量名，禁止引入新的头文件依赖；（2）注入函数调用（HAL\_BE\_* 系列）必须符合函数签名规范；（3）必须保留所有非硬件相关的业务逻辑（如状态字段更新、计数器维护、缓冲区指针操作）；（4）函数签名必须与原函数完全一致，包括返回类型、参数类型和参数名。

\textbf{输出格式}。提示词要求 LLM 以结构化的 JSON 格式输出分类结果，包含函数名、分类类别、分类理由、替换策略和替换代码（仅替换类函数需要）。结构化输出便于后续模块自动解析和处理，避免了自由格式文本解析的不确定性。

\subsubsection{修复提示词模板设计}

闭环修复智能体的提示词模板针对运行时错误修复场景设计，其结构与分类提示词有所不同。修复提示词的核心特点是将错误诊断和修复方案生成整合在一个推理流程中。

\textbf{错误信息结构化}。修复提示词的输入包含结构化的错误信息，包括错误类型（崩溃、超时、内存异常）、错误地址（程序计数器值）、相关函数调用栈、MMIO 函数调用日志和仿真运行的标准输出/标准错误。提示词指导 LLM 首先根据错误类型选择诊断策略：崩溃错误通过 PC 地址匹配定位到具体函数；超时错误通过 PC 分布直方图识别循环结构；内存异常通过访问地址判断是否为未替换的 MMIO 操作。

\textbf{修复约束}。修复提示词设置了严格的修复约束：（1）每轮修复仅针对诊断出的首个问题函数进行修改，避免同时修改多个函数导致冲突；（2）修复仅修改替换代码，不修改原始固件代码；（3）修复必须说明修改原因和预期效果，便于故障追溯；（4）修复后需要保留原替换代码的版本记录，支持回退操作。

\textbf{工具调用策略}。修复提示词引导 LLM 在修复前先通过工具调用获取充分的上下文信息：调用 GetFunctionInfo 获取当前替换代码、调用 GetMMIOFunctionEmulateInfo 获取 MMIO 函数的仿真调用日志、调用 GetFunctionCallsEmulateInfo 获取函数调用栈。这种"先查询再修复"的策略避免了 LLM 在信息不充分的情况下做出错误决策。

\section{动态测试反馈闭环模块实现}

本节阐述动态测试反馈闭环模块的具体实现，其设计方案见第~\ref{chap:design}~章第~3.3~节。

\subsection{仿真平台实现}

\subsubsection{Unicorn 引擎初始化与内存映射}

系统在初始化阶段通过 Unicorn 引擎的 C API 完成仿真环境的搭建。首先调用 \texttt{uc\_open(UC\_ARCH\_ARM, UC\_MODE\_THUMB, \&uc)} 创建 ARM Thumb 模式的模拟器实例，随后依次完成以下配置：（1）内存区域映射，根据目标芯片的链接脚本解析 Flash、RAM 和外设区域的起始地址与大小，调用 \texttt{uc\_mem\_map()} 为各区域分配独立的内存空间并配置读写权限，系统定义了 flash、ram、peripheral\_ram 等受保护的内存类型，防止不同区域被错误合并；（2）固件映像加载，支持 ELF 和 BIN 两种格式，通过自动识别代码段（而非数据段）作为 Flash 基址，调用 \texttt{uc\_mem\_write()} 将固件二进制写入虚拟内存；（3）寄存器初始化，通过 \texttt{uc\_reg\_write()} 设置程序计数器（PC）和栈指针（SP）等关键寄存器的初始值，确保固件从正确的入口点开始执行；（4）浮点单元支持，为兼容包含浮点运算的固件（如图像处理、网络协议栈），启用 VFP 浮点单元支持。

\subsubsection{钩子机制的实现}
\label{sec:hook-mechanism}

第~\ref{chap:design}~章阐述了本系统使用的四类钩子（代码钩子、基本块钩子、内存钩子、中断钩子）的设计原理，本节阐述其具体实现。

在钩子注册阶段，系统通过 \texttt{uc\_hook\_add()} 函数批量注册四类钩子，每类钩子绑定独立的 Python 回调函数。各回调函数接收 Unicorn 实例、触发地址和访问参数，将事件信息写入结构化的日志队列。具体实现中，代码钩子通过 \texttt{UC\_HOOK\_CODE} 类型注册，回调函数接收当前指令地址和指令大小，用于指令计数和执行轨迹记录；基本块钩子通过 \texttt{UC\_HOOK\_BLOCK} 类型注册，回调函数记录基本块入口地址，用于覆盖率统计；内存钩子通过 \texttt{UC\_HOOK\_MEM\_READ} 和 \texttt{UC\_HOOK\_MEM\_WRITE} 类型注册，回调函数接收访问地址、访问大小和数据内容，用于检测对未映射地址空间的非法访问；中断钩子通过 \texttt{UC\_HOOK\_INTR} 类型注册，回调函数接收中断号，用于处理 SVC 指令等软件中断。

在与驱动替换策略配合时，钩子机制承担了关键的拦截角色。系统在仿真初始化阶段通过 ELF 符号表解析所有 \texttt{HAL\_BE\_*} 注入函数的虚拟地址，然后在这些地址处注册代码钩子。当固件执行到注入函数入口时，钩子回调函数执行以下流程：（1）从 Unicorn 寄存器中读取函数参数（ARM 调用约定下参数通过 r0-r3 寄存器传递）；（2）调用宿主机侧的 Python handler 接口完成数据注入或输出操作；（3）将返回值写入 r0 寄存器；（4）从栈中读取返回地址并设置 PC 指针，跳过原函数体继续执行。通过这一机制，固件中调用 \texttt{HAL\_BE\_*} 系列接口时，执行流被拦截并转移到宿主机侧的处理例程。

\subsubsection{弱符号桩函数实现}

弱符号桩函数是连接固件代码和仿真环境的桥梁，其设计原理见第~\ref{chap:design}~章。在工程构建阶段，这些弱符号桩被编译为空函数体，仅作为链接占位符存在，确保固件在没有仿真环境时仍能编译通过。在仿真运行阶段，这些函数的地址被 Unicorn 钩子拦截，实际功能由宿主机侧的 Python 处理器完成。

弱符号桩函数的实现如代码~\ref{lst:weak-stubs}所示。所有桩函数均使用 \texttt{\_\_attribute\_\_((weak))} 修饰，链接器在存在强符号定义时优先使用强符号，否则使用弱符号桩。在仿真配置中，系统通过替换函数机制将这些桩函数的实际执行逻辑重定向到钩子回调中。通过弱符号桩与 Unicorn Hook 的协同机制，固件代码在硬件环境和仿真环境之间可以零修改切换，同一份源码工程无需维护两套代码副本即可同时支持两种运行模式。

\subsubsection{运行日志记录模块实现}

运行日志记录模块通过前述的 Unicorn 钩子机制，将程序执行过程中的关键事件转化为结构化的日志记录。模块在每类钩子回调中将事件信息写入线程安全的日志队列，仿真结束后统一导出为 JSON 格式供故障诊断使用。日志记录包含以下四类信息：（1）指令级信息，利用代码钩子记录指令地址、指令大小和关键寄存器快照；（2）基本块级信息，利用基本块钩子记录每个基本块的入口地址和执行次数，据此构建代码覆盖率统计；（3）函数调用信息，在驱动函数入口和出口地址处设置代码钩子，记录调用地址、被调用函数名、参数值和返回值；（4）中断触发信息，利用中断钩子记录中断号、触发地址和处理器状态。这些日志信息在仿真运行过程中实时采集，为故障诊断和反馈闭环提供运行时数据基础。

\subsection{仿真执行智能体实现}

仿真执行智能体基于 LangGraph 框架实现，其设计见第~\ref{chap:design}~章。在实现层面，该智能体通过 \texttt{StateGraph} 类构建图式状态机，核心状态定义如下：

\begin{lstlisting}[style=Python,language=Python,caption=仿真执行智能体状态定义,label=lst:emulator-state]
@dataclass
class EmulatorState(TypedDict):
    messages: Annotated[list, add_messages]
    project_path: str
    build_config: dict
    replacement_functions: list
    iteration_count: int
    max_iterations: int
    processed_functions: list
    emulate_result: Optional[dict]
\end{lstlisting}

智能体的图结构包含以下节点：\texttt{emulate}（仿真运行节点，调用 EmulateProject 工具）、\texttt{diagnose}（故障诊断节点，调用 GetMMIOFunctionEmulateInfo 和 GetFunctionCallsEmulateInfo 工具）、\texttt{fix}（修复节点，调用 Fixer 子图）、\texttt{build}（构建节点，调用 Builder 子图）。节点间的条件转移逻辑根据仿真结果决定：仿真成功时直接终止；失败时进入诊断节点；诊断完成后进入修复节点；修复完成后进入构建节点；构建完成后返回仿真节点进行回归验证。每个节点的执行由 LLM 驱动决策，LLM 根据当前状态和工具返回的结果决定下一步操作。

\subsection{反馈闭环实现}

\subsubsection{多智能体协同的 LangGraph 实现}

第~\ref{chap:design}~章设计了由仿真执行智能体、修复智能体和构建智能体组成的多智能体架构，本节阐述其基于 LangGraph 的具体实现。

三个智能体通过 LangGraph 的子图（Subgraph）机制集成。仿真执行智能体作为主图，修复智能体和构建智能体分别编译为独立的 \texttt{CompiledGraph} 对象，通过工具调用接口被主图节点调用。这种设计使得各智能体拥有独立的状态空间和工具链，同时通过标准化的输入输出接口实现数据交换。

修复智能体接收仿真执行智能体传递的结构化输入，包括标准输出、标准错误、MMIO 函数调用日志和函数调用栈信息，输出为 \texttt{ReplacementUpdate} 对象（包含函数名、替换代码和修复原因）。构建智能体接收修复智能体的输出，执行代码应用和编译构建，输出为 \texttt{BuildOutput} 对象（包含退出码、编译输出和错误信息）。所有智能体间通信均通过预定义的数据结构进行，避免自由格式文本传递导致的理解偏差。

\subsubsection{闭环工作流的具体实现}

第~\ref{chap:design}~章给出了闭环协同工作流的核心逻辑，本节阐述其实现中的关键技术细节。

在故障诊断环节，系统实现了基于程序计数器（PC）分析的崩溃定位方法。当仿真运行崩溃时，系统从 Unicorn 引擎获取崩溃时刻的 PC 值，将其与静态分析模块输出的函数地址范围进行匹配，定位崩溃发生的具体函数。对于超时类型的故障，系统分析超时时刻的 PC 分布直方图，识别反复执行的指令区间和对应的循环结构，判断是否由未正确替换的轮询函数引起。

在修复执行环节，修复智能体采用"锁定单函数"策略：每轮修复仅针对诊断出的首个问题函数进行修改，避免同时修改多个函数导致修复方案相互冲突。修复策略包括：为 LOOP 类函数添加超时退出条件、补充遗漏的状态字段更新、将不适配的 Skip 策略调整为 Redirect 策略等。每次修复后，系统通过数据版本管理机制保留替换代码的历史版本，当修复导致状态恶化时支持快速回退。

在回归验证环节，系统在每次构建成功后自动重新生成仿真配置（包括内存映射、PC/SP 初始值和钩子注册信息），然后执行完整的仿真运行。系统设置最大迭代次数限制（默认值为 10，与第~\ref{chap:design}~章设计一致），当达到限制后终止闭环并生成包含每轮迭代失败现象和修复方案的详细故障报告。

\subsubsection{终止条件与错误处理的实现}

闭环的终止判断通过 LangGraph 的条件边（conditional edge）实现。在仿真节点之后，条件函数检查仿真结果：若 \texttt{emulate\_result.success == true}，转移到终止节点；否则检查迭代计数是否达到上限、诊断是否能定位问题函数，决定继续迭代或终止。

系统实现了三层错误处理机制：（1）编译错误处理，由构建智能体在编译失败时触发，错误信息通过正则表达式解析后传递给修复智能体；（2）运行时错误处理，包括内存访问错误（UC\_ERR\_FETCH\_UNMAPPED、UC\_ERR\_WRITE\_UNMAPPED）、非法指令执行（UC\_ERR\_INSTRUCTION\_INVALID）和超时，错误信息从 Unicorn 异常中提取后传递给故障诊断模块；（3）框架级错误处理，当 LangGraph 达到递归深度限制（GraphRecursionError）时，系统捕获异常并分析对话历史，生成故障报告后退出。系统记录每次迭代的完整信息（失败现象、诊断结果、修复方案、验证结果），便于问题追溯和系统优化。

\section{实现挑战与解决方案}

在系统实现过程中，本研究面临多项技术挑战，本节对这些挑战及其解决方案进行总结和讨论。

\subsection{CodeQL 对嵌入式 C 代码的支持问题}

\textbf{挑战}。CodeQL 的 C/C++ 分析引擎主要面向通用软件项目设计，对嵌入式 C 代码的某些语言特性支持不够完善，具体表现在以下方面：（1）编译器扩展语法，如 ARM GCC 的 \texttt{\_\_attribute\_\_}、\texttt{\_\_IO}（volatile 修饰）、\texttt{\_\_weak}（弱符号）等关键字，CodeQL 解析器可能无法正确识别，导致语法树构建失败；（2）宏展开问题，嵌入式固件大量使用条件编译宏（如 \texttt{\#ifdef STM32F401xx}）和寄存器访问宏（如 \texttt{\#define USART1 ((USART\_TypeDef *)USART1\_BASE)}），部分宏展开后的表达式超出 CodeQL 的类型推断能力；（3）汇编内嵌代码，部分底层驱动函数包含内联汇编（如 \texttt{\_\_ASM volatile("cpsie i")}），CodeQL 无法分析汇编代码中的数据流。

\textbf{解决方案}。针对上述问题，本研究采取了以下应对策略：（1）预处理适配，通过自定义编译参数向 CodeQL 提供正确的芯片型号宏定义（如 \texttt{-DSTM32F401xx}），确保条件编译分支的正确展开；对于 \texttt{\_\_IO} 等嵌入式扩展关键字，通过在 CodeQL 配置中添加类型映射规则，将其映射为标准的 \texttt{volatile} 修饰符；（2）宏展开降级处理，对于 CodeQL 无法展开的复杂宏，采用保守策略将其标记为未知表达式，不影响宏所在函数的整体识别；（3）汇编代码过滤，在查询规则中通过文件路径和函数签名过滤包含内联汇编的函数，将其标记为需人工确认的候选驱动函数，而非直接跳过。

\subsection{LLM 生成代码的可靠性保障}

\textbf{挑战}。大语言模型生成的代码可能存在以下可靠性问题：（1）语法错误，如括号不匹配、缺少分号、关键字拼写错误等，这类错误虽简单但会阻断编译流程；（2）签名偏差，LLM 可能将函数参数类型误写（如将 \texttt{void *} 写为 \texttt{uint8\_t *}）或遗漏参数，导致接口不一致；（3）语义偏差，替换代码虽然在语法上正确且签名匹配，但语义行为与原函数不一致，如遗漏必要的状态更新、错误的返回值等，这类问题无法通过静态检查发现；（4）幻觉问题，LLM 可能引用不存在的结构体成员或函数，或编造不存在的 API 调用。

\textbf{解决方案}。本研究通过多层防护机制保障生成代码的可靠性：（1）四层验证机制（详见第~\ref{chap:design}~章第~3.2.3~节），在代码写入工程前通过语法检查、签名匹配、返回值检查和替换规则检查四道关卡过滤不合格代码；（2）编译闭环修复，即使验证通过的代码仍可能因芯片型号差异导致编译失败（如不同 STM32 型号的 HAL 结构体成员不同），编译修复智能体通过解析编译错误并针对性修复解决这一问题；（3）动态验证反馈，通过在仿真环境中实际运行替换后的固件，检测语义偏差和运行时错误，触发闭环修复；（4）提示词约束，在 LLM 的分类和生成提示中明确约束只能使用原函数中出现的类型和变量名，禁止引入新的依赖（详见第~\ref{sec:prompt-engineering}~节），从源头减少幻觉问题。

\subsection{Unicorn 仿真环境搭建与 Hook 机制实现}

\textbf{挑战}。Unicorn 作为指令级 CPU 模拟器，仅提供基础的指令集模拟和内存管理能力，不包含外设模拟。在固件仿真场景中需要解决两个核心问题：（1）地址空间配置——不同 MCU 平台的 Flash、SRAM 和外设寄存器区域的地址范围各不相同，需要准确解析链接脚本并配置 Unicorn 的内存映射，否则固件无法正常加载和执行；（2）注入接口对接——替换代码中调用的 \texttt{HAL\_BE\_*} 系列注入函数需要在仿真环境中被正确拦截并由宿主机侧执行，这要求在固件地址空间与宿主机处理逻辑之间建立可靠的桥接机制。

\textbf{解决方案}。在地址空间配置方面，系统根据目标芯片的链接脚本自动解析 Flash 和 SRAM 的起始地址与大小，一次性完成内存映射和固件映像加载，并通过设置程序计数器（PC）和栈指针（SP）的初始值确保固件从正确的入口点开始执行（详见第~\ref{sec:hook-mechanism}~节）。在注入接口对接方面，系统通过 ELF 符号表解析注入函数的虚拟地址并在该地址处注册代码钩子，当执行流到达注入函数入口时，钩子回调从寄存器中读取参数、调用宿主机侧 Python 处理器、将返回值写回寄存器，然后跳过原函数体继续执行。这一机制的实现细节见第~\ref{sec:hook-mechanism}~节。

\subsection{多智能体状态管理与协同}

\textbf{挑战}。系统采用多个智能体（分析智能体、编译修复智能体、仿真智能体、闭环修复智能体）协同工作，智能体之间的状态同步和协同调度面临以下挑战：（1）状态一致性，各智能体通过共享的数据管理器读写替换代码和构建状态，并发访问可能导致数据不一致；（2）错误传播，一个智能体的错误输出可能被下游智能体放大，如分析智能体的错误分类导致生成无效替换代码，进而触发多轮无效的编译修复；（3）资源竞争，多个智能体同时访问编译工具链和仿真引擎可能导致资源竞争和死锁。

\textbf{解决方案}。本研究通过以下机制保障多智能体协同的可靠性：（1）串行化调度，虽然 LangGraph 框架支持并行执行，但系统在关键路径上采用串行调度策略，确保同一时刻只有一个智能体修改工程文件或触发编译，避免并发冲突；（2）数据版本管理，数据管理器为每次修改维护版本记录，支持回滚到任意历史版本，当修复导致状态恶化时可快速回退；（3）步数限制与熔断机制，每个智能体的工具调用次数设置上限（如分析智能体最多 15 步、编译修复智能体最多 10 步），超过上限时触发熔断，由上层调度器决定终止或重试；（4）结构化通信协议，智能体之间通过预定义的数据结构（如 \texttt{ReplacementUpdate}、\texttt{BuildResult}、\texttt{EmulateResult}）交换信息，避免自由格式的自然语言传递导致的理解偏差。

\subsection{系统性能与可扩展性分析}

本节对系统的各阶段性能进行分析，识别性能瓶颈并讨论可扩展性。

\textbf{静态分析阶段}。CodeQL 数据库构建的时间取决于固件工程的规模，典型嵌入式固件工程（含 HAL 库，约 5 万行 C 代码）的数据库构建时间约为 30--60 秒，查询执行时间约为 5--15 秒。静态分析阶段的总时间主要由数据库构建决定，且仅需执行一次，分析结果通过 JSON 缓存可供后续模块重复使用。

\textbf{替换代码生成阶段}。该阶段的耗时主要取决于 LLM 推理延迟和待替换函数的数量。对于单个驱动函数，分析智能体的分类和生成过程通常需要 2--3 次 LLM 调用（信息获取、分类、生成），每次调用的延迟约为 3--10 秒（取决于使用的 LLM 服务和函数代码长度）。对于包含 30--50 个驱动函数的典型固件工程，替换代码生成阶段的总时间约为 5--15 分钟。该阶段的性能瓶颈在于 LLM 推理延迟，可通过批量处理和并行调用（对不同函数同时发起 LLM 请求）来优化。

\textbf{编译构建阶段}。交叉编译的时间取决于固件工程规模和编译工具链性能，典型工程的完整构建时间约为 10--30 秒。在编译失败时，构建修复智能体需要进行额外的 LLM 调用和重新编译，单次修复迭代增加约 20--40 秒的开销。

\textbf{仿真运行阶段}。基于 Unicorn 的仿真运行时间取决于固件的执行路径长度和配置的超时上限。典型固件的仿真运行时间为 5--60 秒，其中指令级钩子带来的性能开销约为 2--5 倍（相比无钩子的 Unicorn 执行）。在动态反馈闭环中，每轮修复迭代（诊断+修复+构建+仿真）的总时间约为 1--3 分钟，系统设置的最大迭代次数为 10 轮。

\textbf{可扩展性}。系统当前的可扩展性限制主要来自三个方面：（1）CodeQL 对超大型固件工程（超过 50 万行代码）的数据库构建时间显著增加，可通过增量分析缓解；（2）LLM 服务的并发请求限制影响大批量函数的处理速度，可通过队列化和速率控制解决；（3）Unicorn 仿真仅支持单核 CPU 模拟，无法仿真多核 MCU 固件，这是轻量级 CPU 模拟器的固有局限。

\section{本章小结}

本章详细阐述了基于驱动程序替换的固件仿真系统的具体实现细节。实现工作围绕第三章提出的设计方案展开，采用 CodeQL、LangGraph、Unicorn 等技术栈，构建了完整的自动化工具链。

在静态分析模块实现方面，本研究基于 CodeQL 引擎实现了 MMIO 访问模式识别、驱动函数定位和数据流追踪等功能。设计了针对三种 MMIO 定位模式的查询规则，构建了统一的数据模型，实现了从源码到硬件依赖信息的自动提取。

在替换模块实现方面，本研究基于 LangGraph 框架实现了多智能体架构。分析智能体负责函数分析和替换代码生成，编译修复智能体负责代码集成和编译，闭环修复智能体负责错误分析和修复。通过结构化的提示词工程（包括分类提示词模板和修复提示词模板）和替换策略实现，系统能够自动生成高质量的替换代码。

在动态反馈模块实现方面，本研究基于 Unicorn 引擎构建了仿真平台。通过弱符号桩函数和 Python 处理器，实现了固件与仿真环境的数据交互。通过多智能体协同工作和闭环迭代优化，系统能够自动诊断和修复运行时问题，显著减少了人工干预。

在实现挑战方面，本章讨论了 CodeQL 对嵌入式 C 代码的支持问题及预处理适配方案、LLM 生成代码的多层可靠性保障机制、Unicorn 仿真环境搭建与 Hook 机制实现以及多智能体状态管理与协同调度方案。这些解决方案的积累为后续系统的可扩展性和稳定性提供了工程基础。

实现结果表明，系统能够在无人干预的情况下完成从静态分析到动态反馈的完整流程。
