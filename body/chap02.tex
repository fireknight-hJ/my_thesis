本章介绍了物联网设备固件仿真研究涉及的基础知识，包括物联网设备硬件基础、物联网操作系统与驱动框架、大模型代码分析技术以及固件仿真与模糊测试技术，为后续章节的论述奠定基础。

\section{物联网设备硬件基础}
\label{sec:hw}

物联网设备通常采用嵌入式微控制器（Microcontroller Unit, MCU）作为核心处理器。常见的 MCU 架构包括 ARM Cortex-M 系列、RISC-V 等。这些处理器具有低功耗、低成本、实时性强等特点，广泛应用于物联网领域。本节重点介绍 ARM Cortex-M 架构的特性以及内存映射 I/O 机制。

\subsection{ARM Cortex-M 架构特性}
\label{sec:cortex_m}

ARM Cortex-M 是 ARM 架构处理器核心中低端系列的统称，由 ARM 公司所授权。这组核心的特点为低成本以及高能源效率的微处理器优化设计，已有上千万个消费性设备中有此系列的微处理器。ARM Cortex-M 系列微处理器采用了 Thumb-2 指令集，提供了 32 位的性能和 16 位的代码密度。同时该系列提供了一些特殊的功能，如嵌套向量中断控制器（Nested Vectored Interrupt Controller, NVIC）、系统计时器（System Tick Timer, Systick）、位带存储器（Bit-Band Memory）、存储器保护单元（Memory Protect Unit, MPU）等，可以提高系统的可靠性和实时性。

\subsubsection{嵌套向量中断控制器（NVIC）}

NVIC 是 ARM Cortex-M 架构的一个核心组件，负责处理外部和内部的中断请求。嵌入式操作系统可以通过配置 NVIC 的优先级寄存器和使能寄存器，来设置不同中断源的优先级和使能状态，从而实现中断的响应和屏蔽。同时，操作系统可以通过读取 NVIC 的挂起寄存器和活动寄存器，来判断当前是否有中断发生，以及哪些中断正在执行。这些信息对于任务调度和中断嵌套是非常重要的。

\subsubsection{系统定时器（Systick）}

Systick 是 ARM Cortex-M 架构的另一个核心组件，负责提供一个可编程的周期性中断。嵌入式操作系统通过配置 Systick 的重装载寄存器和控制寄存器，来设置 Systick 的时钟源、计数模式和中断使能。通常，操作系统会将 Systick 设置为产生一个固定频率的中断，比如 1ms 或 10ms，作为操作系统的心跳。这样，操作系统就可以利用 Systick 中断来实现时间管理和任务切换等功能。

\subsubsection{栈模式切换}

主栈与进程栈是 ARM Cortex-M 架构支持的两种栈模式。主栈是默认的栈模式，用于执行异常处理程序（包括复位、NMI、硬件错误和 SVC 等）。进程栈是可选的栈模式，用于执行线程模式下的用户程序。操作系统可以通过配置控制寄存器的 SPSEL 位，来选择当前使用哪种栈模式。栈模式的切换是实现多任务并发和内存保护的基础。

\subsection{内存映射 I/O}
\label{sec:mmio}

内存映射 I/O（Memory-Mapped I/O, MMIO）是嵌入式系统中外设与处理器通信的主要方式。与端口映射 I/O（Port-Mapped I/O, PMIO）不同，MMIO 不需要专用的 I/O 指令，而是将外设寄存器映射到处理器的统一内存地址空间中。处理器通过普通的内存读写指令（如 \texttt{LDR}/\texttt{STR}）访问特定内存地址，即可完成对外设的控制和数据传输。这种设计简化了编程模型，使得外设访问与内存访问具有相同的语法形式。

\subsubsection{MMIO 地址空间布局}

ARM Cortex-M 架构采用统一的 32 位地址空间，理论上可寻址 4GB 内存。虽然具体的地址映射可能因芯片厂商和型号而有所不同，但 ARM 定义了推荐的内存映射框架，将地址空间按功能划分为若干区域。

代码区位于地址空间的低端，用于存储程序指令和常量数据。该区域支持指令取指操作，通常映射到 Flash 存储器或其他非易失性存储器。SRAM 区紧随其后，用于存储变量、堆栈和动态分配的内存。部分 Cortex-M 处理器支持位带（Bit-Band）操作，允许对单个位进行原子访问，这在嵌入式编程中非常有用。

外设区是 MMIO 的核心区域，所有片上外设的寄存器都被映射到此区域。不同厂商的外设被分配到不同的地址段，例如某厂商的 GPIO 可能位于某一地址范围，而 USART 位于另一地址范围。这种映射使得处理器可以通过标准的内存访问指令来控制外设。

系统区位于地址空间的高端，包含处理器核心相关的调试接口和系统控制寄存器。嵌套向量中断控制器（NVIC）、系统定时器（SysTick）、内存保护单元（MPU）等系统级组件的控制寄存器都位于此区域。此外，系统区还包含厂商特定的系统配置寄存器。

理解 MMIO 地址空间布局对于固件仿真至关重要。仿真器需要正确处理对不同区域的访问：对于代码区和 SRAM 区，可以采用标准的内存模拟；对于外设区和系统区，则需要根据具体外设的行为实现专门的访问处理逻辑。

\subsubsection{外设寄存器分类}

外设寄存器按照功能可分为控制寄存器、状态寄存器和数据寄存器三类。控制寄存器（Control Register）用于配置外设的工作模式和参数，通常是可读写的，固件通过写入特定值来设置外设功能。例如，USART 的控制寄存器 CR1 可以设置波特率、数据位长度、校验位等参数。控制寄存器的写入操作通常不会立即产生硬件响应，而是配置外设在后续操作中的行为。

状态寄存器（Status Register）反映外设的当前工作状态，通常是只读的，固件通过读取状态寄存器来判断外设是否就绪、是否发生错误等。例如，USART 的状态寄存器 SR 包含 TXE（发送缓冲区空）、RXNE（接收缓冲区非空）等标志位。状态寄存器的值由硬件根据外设实际状态自动更新，这是 MMIO 与普通内存访问的关键区别。

数据寄存器（Data Register）用于数据传输，通常是可读写的，写入操作将数据发送到外设，读取操作从外设接收数据。例如，USART 的数据寄存器 DR 用于发送和接收串口数据。对数据寄存器的访问通常会触发外设执行相应操作（如开始数据传输），并可能改变状态寄存器的值。

\subsubsection{MMIO 在固件仿真中的挑战}

MMIO 机制给固件仿真带来了独特的技术挑战，这些挑战源于硬件外设的复杂性和 MMIO 访问模式的特殊性。

\textbf{（1）寄存器语义的多样性与不确定性}

不同类型的外设具有完全不同的寄存器布局和行为语义。例如，UART 外设的数据寄存器用于串口通信，而 GPIO 外设的数据寄存器用于控制引脚电平。即使是同一类型的外设，不同厂商的实现也可能存在差异，寄存器数量、位域定义、操作时序都可能不同。这种多样性使得难以构建通用的外设仿真模型。更复杂的是，许多嵌入式固件使用厂商自定义的外设，这些外设的文档可能不完整甚至完全缺失，导致仿真器难以准确理解寄存器的语义。

\textbf{（2）状态寄存器的动态响应问题}

状态寄存器的值不是固定的，而是根据外设的内部状态动态变化的。例如，UART 的状态寄存器中的接收缓冲区非空（RXNE）标志位，只有当外设实际接收到数据时才会置位。在仿真环境中，如果没有真实的数据输入，这个标志位应该如何响应？简单的静态返回值无法准确模拟这种行为，而完全模拟外设的状态机又需要深入理解外设的工作原理。这种状态依赖性使得固件可能因为等待某个永远不会出现的状态而无限循环。

\textbf{（3）寄存器访问的副作用}

对 MMIO 寄存器的某些访问会产生副作用，这是 MMIO 与普通内存访问的关键区别。例如，读取 UART 的数据寄存器不仅返回接收到的字节，还会自动清除接收缓冲区非空标志；写入中断清除寄存器的某些位会清除对应的中断挂起状态。如果仿真器不能正确模拟这些副作用，固件的执行逻辑就会出现错误。识别和理解这些副作用需要仔细分析外设文档，或者通过动态观察真实硬件的行为来推断。

\textbf{（4）多寄存器操作的原子性与时序约束}

许多外设操作涉及多个寄存器的协调访问。例如，配置一个外设可能需要按特定顺序写入多个控制寄存器，如果顺序错误可能导致配置失败或外设状态异常。某些外设对时序有严格要求，如在设置使能位后必须等待一定周期才能访问数据寄存器。这些时序约束在简单的寄存器读写模拟中难以表达，但在真实硬件上是必须遵守的。仿真器需要在保持足够精度的同时，又不能过度限制以避免影响性能。

这些挑战使得传统的内存模拟方法无法直接应用于 MMIO 区域，需要针对不同外设实现专门的仿真逻辑。正是这些复杂性，驱动了基于驱动层替换的仿真方法——通过识别和替换访问 MMIO 的驱动函数，可以在函数调用层面进行抽象，避免直接模拟底层硬件的复杂行为。

这些挑战使得传统的内存模拟方法无法直接应用于 MMIO 区域，需要针对不同外设实现专门的仿真逻辑。这正是驱动层替换方法的核心动机——通过识别和替换访问 MMIO 的驱动函数，避免直接模拟复杂的硬件行为。

\section{物联网操作系统与驱动}
\label{sec:os}

\subsection{实时操作系统概述}
\label{sec:rtos}

实时操作系统（Real-Time Operating System, RTOS）是物联网设备中广泛采用的管理任务调度和资源分配的软件系统。RTOS 能够及时响应外部事件或数据，并在规定的时间内完成任务，其工作原理是通过一个实时任务调度器来管理系统资源，按照优先级或者时间片来分配 CPU 时间给不同的任务，保证重要或者紧急的任务优先执行。

物联网领域常见的 RTOS 包括 FreeRTOS\cite{freertos}、Zephyr\cite{zephyr}、RT-Thread\cite{rtthread} 等。FreeRTOS 是 Amazon 开源的实时操作系统，专为微控制器和小型微处理器设计，内核精简，最小可运行在不到 10KB 的 RAM 上，是物联网领域使用最广泛的 RTOS 之一。Zephyr 是 Linux 基金会管理的开源实时操作系统，支持多种处理器架构，提供更完整的操作系统功能，包括文件系统、网络协议栈、设备驱动模型等，采用模块化设计以平衡功能丰富性和资源占用。RT-Thread 是中国开源社区主导开发的实时操作系统，采用微内核架构，核心组件包括线程调度、线程间通信、内存管理、设备驱动等，并提供了丰富的组件生态。这些 RTOS 都提供了设备驱动框架，为不同类型的外设定义了统一的操作接口，降低了应用开发的复杂度。

\subsection{硬件抽象层}
\label{sec:hal}

硬件抽象层（Hardware Abstraction Layer, HAL）是嵌入式软件架构中的关键组成部分，它位于硬件驱动层和上层应用之间，为应用程序提供统一的硬件访问接口。HAL 层的核心目标是屏蔽底层硬件的差异性，使得上层应用代码可以在不同的硬件平台之间移植，而无需修改业务逻辑。

在传统的嵌入式开发中，应用程序需要直接操作寄存器来完成外设控制，这种方式不仅代码可读性差，而且当更换硬件平台时需要大量修改。HAL 层通过定义统一的接口规范，将硬件操作封装为函数调用，显著提高了代码的可维护性和可移植性。例如，不同厂商的 UART 外设寄存器布局可能完全不同，但通过 HAL 层，应用程序可以使用统一的 \texttt{HAL\_UART\_Transmit()} 函数来发送数据，而无需关心底层寄存器的具体实现。

HAL 层的接口设计通常遵循一定的命名和功能规范。以 STM32 的 HAL 库为例，每个外设类型都有一组标准化的接口函数：初始化函数（如 \texttt{HAL\_UART\_Init}）负责配置外设参数和使能时钟；数据传输函数（如 \texttt{HAL\_UART\_Transmit}、\texttt{HAL\_UART\_Receive}）封装了数据的发送和接收逻辑；状态查询函数（如 \texttt{HAL\_UART\_GetState}）用于获取外设的当前工作状态；中断管理函数（如 \texttt{HAL\_UART\_Receive\_IT}）用于配置中断模式和注册回调函数。这种标准化的接口设计使得开发者可以快速熟悉不同外设的使用方法。

HAL 层的抽象特性为固件仿真提供了重要基础。由于 HAL 函数封装了具体的硬件操作细节，仿真器可以通过替换 HAL 层函数来实现硬件行为的模拟，而无需深入理解底层寄存器的语义。这种函数级的替换相比寄存器级的模拟更加高效，也更容易实现。在实际的固件仿真系统中，通过识别和替换 HAL 层的硬件相关函数，可以有效解耦固件逻辑与硬件依赖，使固件能够在没有真实硬件的环境中正常运行。

\section{大模型代码分析智能体框架}
\label{sec:llm}

大语言模型（Large Language Model, LLM）在近年来取得了突破性进展，其强大的代码理解和生成能力为自动化程序分析提供了新的技术途径。本节简要介绍大语言模型的发展历程，重点阐述通用大模型的代码能力以及基于 LLM 的智能体开发框架，为后续章节的驱动函数分析和替换工作奠定技术背景。

\subsection{大语言模型发展概述}

大语言模型的发展始于 2017 年 Transformer 架构的提出。Transformer 完全基于自注意力机制，摒弃了传统的循环神经网络结构，能够高效地建模长距离依赖关系。这一架构创新为后续的大规模预训练语言模型奠定了基础。

以 GPT 系列为代表的仅解码器（Decoder-only）架构通过大规模无监督预训练，展现出强大的语言理解和生成能力。随着模型规模从数亿参数扩展到数千亿参数，这些模型逐渐涌现出少样本学习（Few-shot Learning）、思维链推理（Chain-of-Thought Reasoning）等高级能力。特别是 GPT-4、Claude-3 等新一代模型，不仅在自然语言处理任务上表现优异，在代码生成、程序理解、调试辅助等编程相关任务上也展现出接近甚至超越人类专家的能力。

近年来，开源大模型生态也快速发展。LLaMA、Mistral、Qwen 等开源模型在多个基准测试中取得了与闭源模型相当的性能，为学术研究和工业应用提供了更多选择。这些开源模型的出现，使得在本地部署和定制化训练大语言模型成为可能，降低了技术门槛。

\subsection{通用大模型的代码能力}

大语言模型的代码生成与理解能力，近年来取得了突破性进展，已成为衡量模型性能的核心指标之一。从最初的简单代码补全，到如今能够完成复杂项目开发、跨文件代码重构、技术债务分析等高级任务，大模型的编程能力正以前所未有的速度迭代升级。代码数据因其逻辑严谨、结构清晰的特点，本身也是提升模型推理能力的重要训练素材——一个模型如果擅长写代码，往往在数学、逻辑推理等任务上也表现优异。正因如此，主流大模型厂商纷纷将代码能力作为重点突破方向，推动模型从"能写简单函数"向"理解完整项目"演进。本研究主要使用以下通用大语言模型。

DeepSeek-V3.2 是深度求索公司发布的新一代大语言模型，在代码和推理能力上达到了全球领先水平。该模型采用 DSA（DeepSeek Sparse Attention）稀疏注意力机制，在保持无损长文本性能的同时大幅降低了模型部署成本。DeepSeek-V3.2 的核心创新在于将思考推理融入工具使用，这是首个支持思考模式下工具调用的模型。通过大规模 Agent 训练数据合成方法，模型在 1800 多个环境、85000 多个复杂指令上进行了强化学习训练，显著提高了泛化能力。在公开的推理类基准测试中，DeepSeek-V3.2 达到了 GPT-5 的水平。其长思考增强版 DeepSeek-V3.2-Speciale 更是斩获了 IMO 2025（国际数学奥林匹克）、CMO 2025（中国数学奥林匹克）、ICPC World Finals 2025（国际大学生程序设计竞赛全球总决赛）及 IOI 2025（国际信息学奥林匹克）金牌，其中 ICPC 与 IOI 成绩分别达到了人类选手第二名与第十名的水平。

GLM-5 是智谱 AI 发布的新一代旗舰基础模型，专为 Agentic Engineering 设计，能够在复杂系统工程和长距离 Agent 任务中提供可靠的生产力。GLM-5 将参数规模从 355B（32B 激活）扩展到 744B（40B 激活），预训练数据从 23T 升级到 28.5T，更大规模的预训练计算显著提升了模型的通用智能。模型采用异步强化学习框架 Slime，支持更大模型规模和更复杂强化学习任务，使模型能够从长距离交互中持续学习。在软件工程任务上，GLM-5 实现了与 Claude Opus 4.5 的性能对齐，在 SWE-bench Verified 上取得了 77.8 的开源模型最高分，在 Terminal Bench 2.0 上取得了 56.2 的领先成绩。在智能体能力方面，GLM-5 在 BrowseComp（网页级检索与信息合成）、MCP-Atlas（工具调用与多步任务执行）、τ²-Bench（复杂多工具规划与编排）等多个权威基准测试上均取得了开源模型第一的成绩。

\subsection{LLM 智能体开发框架}

为了更好地利用大语言模型构建复杂的应用系统，研究人员开发了多种智能体（Agent）框架。这些框架提供了结构化的方式来组织 LLM 的调用、管理状态、集成外部工具，使开发者能够构建可靠的、可扩展的 LLM 应用。

\subsubsection{LangChain 框架}

LangChain 是目前最流行的 LLM 应用开发框架之一，提供了丰富的组件和抽象，帮助开发者快速构建 LLM 驱动的应用。LangChain 的核心概念包括模型、提示模板、链、代理、工具和记忆。模型组件封装了与不同 LLM 提供商的交互接口，支持 OpenAI、Anthropic、本地模型等多种后端，开发者可以通过统一的接口调用不同的模型，便于切换和比较。提示模板组件允许开发者定义带变量的提示词模板，在运行时动态填充变量值，使提示词的构造更加灵活和可复用。

链是将多个组件按顺序组合的机制，例如一个代码分析链可能包含读取代码文件、构造提示词、调用 LLM、解析结果、存储分析报告等步骤。通过链的组合，可以构建复杂的处理流程。代理是更高级的抽象，它允许 LLM 自主决定下一步行动。代理根据用户的输入，选择合适的工具（如搜索引擎、代码执行器、数据库查询），执行操作，并根据结果决定后续行动，直到完成任务。LangChain 内置了多种代理类型，如 ReAct 代理、OpenAI Functions 代理等。工具是代理可以调用的外部功能，LangChain 提供了大量内置工具（如网络搜索、Python 代码执行、文件操作等），开发者也可以自定义工具。记忆组件允许应用在多次调用之间保持状态，例如在多轮对话中存储历史对话，使 LLM 能够理解上下文。LangChain 提供了多种记忆实现，如缓冲记忆、摘要记忆、向量存储记忆等。

\subsubsection{LangGraph 状态图}

LangGraph 是 LangChain 生态中用于构建有状态、可循环的智能体工作流的库。与 LangChain 的链式结构不同，LangGraph 基于状态图（State Graph）的概念，支持更复杂的控制流。LangGraph 的核心是状态（State）对象，它在工作流的各个节点之间传递和更新。状态可以包含任意类型的数据，如对话历史、中间结果、用户输入等。每个节点可以读取和修改状态，实现信息的流动和共享。

节点（Node）是工作流中的处理单元，接收状态作为输入，返回状态的更新。边（Edge）定义了节点之间的转换关系。LangGraph 支持条件边（Conditional Edge），根据状态的值动态选择下一个节点，实现分支逻辑。与简单的链式结构不同，LangGraph 支持循环，允许工作流在满足特定条件前重复执行某些节点。这对于迭代优化、多轮交互等场景非常有用。例如，在代码生成任务中，可以设计一个循环：生成代码、编译验证、如果失败则修复、重新生成，直到代码可编译。LangGraph 还支持检查点（Checkpoint）机制，可以将工作流的状态持久化存储，使得工作流可以暂停和恢复，支持长时间运行的任务。在代码分析场景中，如果 LLM 调用超时或失败，可以从上一个检查点恢复，而不需要重新开始。

\subsection{代码分析工作流范式}

基于 LLM 智能体框架，可以构建多种代码分析工作流。ReAct（Reasoning + Acting）是一种结合推理和行动的工作流范式。在代码分析任务中，LLM 首先进行推理（如"我需要理解这个函数的功能"），然后采取行动（如"调用静态分析工具提取函数的调用关系"），观察结果，再进行下一步推理。这种范式使 LLM 能够动态地收集信息，做出决策，逐步完成复杂的分析任务。

Plan-and-Execute 范式将任务分解为规划和执行两个阶段。在规划阶段，LLM 分析任务要求，生成一个执行计划（如识别硬件相关代码、分析驱动函数语义、生成替代代码、验证编译）。在执行阶段，按照计划依次执行各个步骤。这种范式的优点是结构清晰，易于调试和监控。

Multi-Agent 范式使用多个专门的智能体协作完成任务。例如，在驱动函数替换任务中，可以设计分析智能体（负责识别驱动函数，分析其语义和依赖关系）、生成智能体（负责生成替代代码，考虑仿真环境的特点）、验证智能体（负责编译验证，检查代码的正确性）、协调智能体（负责协调各智能体的工作，处理冲突）。各智能体通过消息传递或共享状态进行协作，共同完成复杂的代码分析和生成任务。

反思与迭代范式引入了自我评估和改进机制。在生成初步结果后，LLM 对结果进行反思（如"这段代码是否正确实现了原函数的功能？"），发现潜在问题，然后进行迭代改进。这种范式特别适用于代码生成任务，能够逐步提高生成代码的质量。在本研究的驱动函数替换模块中，采用了类似的迭代机制：生成代码、编译验证、分析错误、修正代码，直到达到可编译状态。这些工作流范式为构建可靠的代码分析系统提供了设计模式。在实际应用中，可以根据任务的特点选择合适的范式，或者组合多种范式，以实现最佳效果。

LangChain 提供了统一的接口来调用不同的大语言模型，包括 OpenAI、Anthropic、智谱AI、深度求索等多个提供商的模型。开发者可以轻松切换不同的模型，而无需修改应用代码。

\textbf{（2）提示词模板（Prompt Templates）}

提示词模板允许开发者定义可复用的提示词结构，通过变量注入实现动态生成。例如，可以定义一个代码分析模板，包含代码片段、分析要求等占位符，在运行时填入具体内容。

\textbf{（3）链（Chains）}

链是将多个组件按顺序组合的机制。例如，一个代码分析链可能包含：读取代码文件 → 构造提示词 → 调用 LLM → 解析结果 → 存储分析报告。通过链的组合，可以构建复杂的处理流程。

\textbf{（4）代理（Agents）}

代理是更高级的抽象，它允许 LLM 自主决定下一步行动。代理根据用户的输入，选择合适的工具（如搜索引擎、代码执行器、数据库查询），执行操作，并根据结果决定后续行动，直到完成任务。LangChain 内置了多种代理类型，如 ReAct 代理、OpenAI Functions 代理等。

\textbf{（5）工具（Tools）}

工具是代理可以调用的外部功能。LangChain 提供了大量内置工具，如网络搜索、Python 代码执行、文件操作等，开发者也可以自定义工具。工具的定义包括名称、描述、输入模式、执行函数等。

\textbf{（6）记忆（Memory）}

记忆组件允许应用在多次调用之间保持状态。例如，在多轮对话中，记忆组件可以存储历史对话，使 LLM 能够理解上下文。LangChain 提供了多种记忆实现，如缓冲记忆、摘要记忆、向量存储记忆等。

\subsubsection{LangGraph 状态图}

LangGraph 是 LangChain 生态中用于构建有状态、可循环的智能体工作流的库。与 LangChain 的链式结构不同，LangGraph 基于状态图（State Graph）的概念，支持更复杂的控制流。

\textbf{（1）状态管理}

LangGraph 的核心是状态（State）对象，它在工作流的各个节点之间传递和更新。状态可以包含任意类型的数据，如对话历史、中间结果、用户输入等。每个节点可以读取和修改状态，实现信息的流动和共享。

\textbf{（2）节点和边}

节点（Node）是工作流中的处理单元，接收状态作为输入，返回状态的更新。边（Edge）定义了节点之间的转换关系。LangGraph 支持条件边（Conditional Edge），根据状态的值动态选择下一个节点，实现分支逻辑。

\textbf{（3）循环结构}

与简单的链式结构不同，LangGraph 支持循环，允许工作流在满足特定条件前重复执行某些节点。这对于迭代优化、多轮交互等场景非常有用。例如，在代码生成任务中，可以设计一个循环：生成代码 → 编译验证 → 如果失败则修复 → 重新生成，直到代码可编译。

\textbf{（4）持久化}

LangGraph 支持检查点（Checkpoint）机制，可以将工作流的状态持久化存储。这使得工作流可以暂停和恢复，支持长时间运行的任务。在代码分析场景中，如果 LLM 调用超时或失败，可以从上一个检查点恢复，而不需要重新开始。

\subsection{代码分析工作流范式}

基于 LLM 智能体框架，可以构建多种代码分析工作流。以下是几种常见的工作流范式：

\textbf{（1）ReAct 范式}

ReAct（Reasoning + Acting）是一种结合推理和行动的工作流范式。在代码分析任务中，LLM 首先进行推理（"我需要理解这个函数的功能"），然后采取行动（"调用静态分析工具提取函数的调用关系"），观察结果，再进行下一步推理。这种范式使 LLM 能够动态地收集信息，做出决策，逐步完成复杂的分析任务。

\section{程序分析技术}
\label{sec:program-analysis}

程序分析技术是软件工程和安全研究的重要基础，旨在通过自动化手段理解程序的行为和性质。在固件仿真研究中，程序分析技术用于识别代码中的硬件依赖、追踪数据流、建立函数调用关系，为后续的驱动函数替换提供准确的"硬件依赖画像"。

\subsection{静态分析与动态分析}

程序分析技术可分为静态分析和动态分析两大类。

静态分析（Static Analysis）在不执行程序的情况下，通过分析源代码或二进制代码来推断程序的属性。静态分析的优势在于能够覆盖程序的所有可能执行路径，不受测试用例的限制；其局限在于面临不可判定性问题（如停机问题），对于某些程序属性只能给出近似答案。常见的静态分析技术包括数据流分析、控制流分析、指针分析、抽象解释等。

动态分析（Dynamic Analysis）通过实际执行程序并观察其运行时行为来理解程序。动态分析能够获取精确的运行时信息（如变量值、内存状态、执行路径），但只能观察到被测试用例覆盖的部分，覆盖率不足可能导致遗漏重要行为。常见的动态分析技术包括动态插桩、污点追踪、符号执行等。

本研究主要采用静态分析技术。对于固件源码而言，静态分析具有以下优势：首先，源码层面的静态分析能够获取完整的程序信息（类型、符号、注释等），分析精度高于二进制层面的分析；其次，静态分析不需要真实硬件或仿真环境，可以在代码开发的早期阶段进行；第三，静态分析能够覆盖所有可能的执行路径，不会遗漏任何驱动函数。

\subsection{CodeQL 静态分析框架}

CodeQL 是 GitHub 开发的语义级代码分析平台，支持多种编程语言（包括 C/C++、Java、Python、JavaScript 等）的安全漏洞检测和代码质量分析。CodeQL 的核心思想是将源代码转换为可查询的数据库，然后通过声明式查询语言（QL）进行灵活的分析。

\subsubsection{CodeQL 数据库构建}

CodeQL 数据库是源代码的结构化表示，包含语法树、控制流图、数据流图等多层次信息。数据库的构建过程分为源码准备、数据库创建和索引建立三个阶段。在源码准备阶段，需要将固件源代码组织为标准项目结构，确保包含所有头文件和依赖。对于使用构建系统（如 CMake、Makefile）的项目，CodeQL 可以利用构建信息进行更精确的分析。在数据库创建阶段，调用 CodeQL 命令行工具创建数据库，指定目标语言、源码路径和构建命令。在索引建立阶段，CodeQL 对 C/C++ 源文件建立多层次索引，包括抽象语法树（AST）表示源代码的语法结构，包含函数定义、变量声明、表达式等语法元素；控制流图（CFG）表示程序执行路径的有向图，表示基本块之间的跳转关系；数据流图（DFG）表示变量和表达式的数据传播关系，用于追踪数据的定义和使用；符号表与类型信息记录变量、函数、类型的定义和引用。

\subsubsection{QL 查询语言}

CodeQL 查询语言（QL）是一种声明式的逻辑编程语言，专门用于代码分析。QL 的语法类似于 Datalog，采用谓词逻辑的表达方式。查询的基本模式是定义实体（如函数、变量、表达式）之间的关系，然后通过逻辑连接构建复杂的分析规则。QL 查询的基本结构包括导入模块（如 \texttt{import cpp} 导入 C/C++ 分析库）、定义谓词（如 \texttt{isMMIOAccess(Expr e)} 判断表达式是否为 MMIO 访问）、定义类（用于封装相关的实体和谓词，支持继承和重写）、选择子句（定义查询的输出格式）。

\subsubsection{数据流分析与污点追踪}

数据流分析是 CodeQL 的核心能力之一，用于追踪数据在程序中的传播路径。污点追踪（Taint Tracking）是数据流分析的一种特殊形式，用于追踪"污点数据"（如用户输入、不可信数据）从源点（source）到汇点（sink）的传播路径。在 CodeQL 中，污点追踪通过配置类（Configuration）定义。源点（Source）定义污点数据的来源，如 MMIO 字段访问、外部输入函数等。汇点（Sink）定义污点数据的影响点，如函数参数、敏感操作等。污点步骤（Taint Step）定义污点数据在程序中的传播方式，如赋值、函数调用、指针解引用等。屏障（Barrier）定义阻断污点传播的语句，如数据清理、边界检查等。通过配置这些要素，CodeQL 能够自动追踪数据流路径，识别潜在的数据流问题。本研究利用 CodeQL 的污点追踪能力，建立 MMIO 访问与函数参数之间的关联关系。

\section{固件仿真与模糊测试}
\label{sec:fuzz}

固件仿真是物联网安全研究的重要技术手段。由于物联网设备的硬件多样性和封闭性，直接在真实设备上进行安全测试面临诸多困难。固件仿真技术通过软件模拟硬件环境，使研究人员能够在通用计算平台上运行和分析嵌入式固件，从而降低测试成本并提高效率。模糊测试作为一种高效的漏洞挖掘技术，与固件仿真相结合，能够在仿真环境中自动化地发现固件中的安全漏洞。

\subsection{QEMU 仿真器}

QEMU\cite{qemu} 是目前应用最广泛的开源机器仿真器，支持多种处理器架构和外设仿真。QEMU 的设计目标是提供高性能、高可移植性的系统仿真能力，使其成为固件仿真研究的重要基础平台。

QEMU 支持两种主要的仿真模式。用户模式仿真（User Mode Emulation）允许在一种架构上运行为另一种架构编译的单个程序，通过动态二进制翻译技术将目标架构的指令转换为宿主机指令执行。用户模式仿真适用于分析用户空间程序，但由于物联网固件通常运行在裸机或嵌入式操作系统上，用户模式仿真在固件仿真领域的应用有限。

系统模式仿真（System Mode Emulation）提供完整的硬件系统仿真，包括处理器、内存、中断控制器、外设等组件。在系统模式下，QEMU 可以运行完整的操作系统或裸机固件，这是固件仿真的主要工作模式。QEMU 的系统模式仿真采用模块化设计，各种硬件组件以独立模块的形式实现，用户可以通过配置文件组合不同的组件来构建目标系统的硬件模型。

QEMU 的核心是动态二进制翻译引擎（Tiny Code Generator, TCG）。TCG 将目标架构的指令块翻译为中间表示（Intermediate Representation, IR），然后再将 IR 翻译为宿主机指令。这种两级翻译机制使 QEMU 能够支持多种目标架构和宿主架构的组合。为了提高性能，TCG 采用即时编译（JIT）技术，将翻译结果缓存在内存中供后续执行重复使用。

QEMU 提供了丰富的外设模型库，涵盖了常见的通信接口（UART、SPI、I2C）、存储设备（Flash、SD 卡）、网络设备（以太网控制器）等。对于物联网设备中常见的外设，QEMU 通常已经有相应的实现。然而，对于厂商自定义的专用外设，研究人员需要基于 QEMU 的外设框架自行实现仿真模型。这需要深入理解外设的硬件规范，工作量较大。

在固件仿真应用中，QEMU 还面临一些挑战。首先是仿真精度与性能的权衡：高精度的仿真需要模拟更多的硬件细节，但会带来显著的性能开销；简化的仿真模型性能更好，但可能与真实硬件行为存在偏差。其次是外设状态的一致性问题：外设的内部状态可能随时间变化，简单的状态机模型难以准确表达这种动态行为。此外，一些物联网设备使用非标准的启动流程和内存映射，需要对 QEMU 进行定制才能正确运行。

\subsection{模糊测试技术}

模糊测试（Fuzzing）是一种自动化的软件测试技术，通过向目标程序输入大量随机或经过变异的数据，触发程序的各种执行路径，从而发现潜在的漏洞和缺陷。模糊测试技术具有自动化程度高、测试覆盖面广、能够发现未知漏洞等优点，在软件安全领域得到了广泛应用。

模糊测试的基本流程包括以下几个步骤：首先，种子选择模块从初始种子集合中选择一个种子作为变异基础；然后，变异模块对种子进行随机变异或基于规则的变异，生成新的测试用例；接着，测试执行模块将测试用例输入目标程序并监控其执行；最后，结果分析模块根据执行结果判断是否触发了异常行为（如崩溃、断言失败等），并将有价值的测试用例加入种子集合供后续变异使用。

模糊测试技术的发展可以分为几个阶段。第一代模糊测试采用黑盒方法，通过生成随机输入数据来测试程序。这种方法实现简单，但由于缺乏对程序结构的了解，测试效率较低，难以触发深层代码路径。第二代模糊测试引入了反馈机制，通过代码插桩等技术获取程序执行路径信息，利用这些信息指导测试用例的生成，代表工具包括 AFL、libFuzzer 等。这种灰盒方法显著提高了测试效率和覆盖率。第三代模糊测试结合了符号执行、污点分析等技术，能够更精确地分析程序约束，生成满足特定路径条件的测试用例，代表工具包括 Driller、QSYM 等。

代码覆盖率是衡量模糊测试效果的重要指标。覆盖率引导的模糊测试（Coverage-guided Fuzzing）在测试过程中持续监控代码覆盖率，优先保留能够触发新执行路径的测试用例，从而逐步扩大测试范围。这种策略使得模糊测试能够在有限的测试时间内探索更多的程序状态，提高漏洞发现的概率。

\subsection{嵌入式固件模糊测试}

嵌入式固件模糊测试是物联网安全研究的重要方向，其核心目标是在仿真环境中对固件进行自动化安全测试。根据测试执行环境的不同，固件模糊测试方法可以分为两类。

\textbf{（1）硬件在环方法}

硬件在环（Hardware-in-the-Loop, HIL）方法在真实硬件上运行固件，通过专用接口（如 JTAG、SWD、串口等）注入测试数据。这种方法能够保证与真实硬件的一致性，测试结果可靠。然而，硬件在环方法也存在明显的局限性：测试速度受限于硬件速度，难以进行大规模并发测试；需要物理访问目标设备，对于已部署的设备难以实施；每次测试后可能需要手动复位设备，自动化程度有限。

\textbf{（2）基于仿真的方法}

基于仿真的方法在软件仿真器（如 QEMU）中运行固件，通过软件接口注入测试数据。这种方法具有测试速度快、易于自动化、可并发执行等优点，是当前研究的主流方向。基于仿真的方法需要解决的核心问题是固件在仿真环境中的硬件依赖：仿真器需要为固件提供足够的外设支持，使固件能够正常运行到测试目标点。针对这一问题，研究人员提出了多种外设建模方法，包括手动建模、基于硬件追踪的自动建模、基于程序分析的推断建模等。

\subsection{AFL 及其变体}

American Fuzzy Lop（AFL）\cite{afl} 是最具影响力的覆盖率引导模糊测试工具之一，其设计思想和技术实现对后续的模糊测试工具产生了深远影响。

AFL 的核心是轻量级的代码插桩和高效的变异策略。在编译时插桩模式下，AFL 通过修改编译器（如 afl-gcc、afl-clang）在程序中插入覆盖率记录代码；在即时插桩模式下，AFL 利用 QEMU 的动态翻译功能在运行时插入插桩代码。无论哪种模式，插桩代码都会记录程序执行过程中经过的基本块和边，这些信息用于计算覆盖率并指导测试用例的选择。

AFL 维护一个测试用例队列，初始包含用户提供的种子文件。在测试过程中，AFL 不断从队列中选择测试用例，对其进行变异，执行变异后的测试用例，并根据覆盖率变化决定是否将新用例加入队列。AFL 的变异策略包括位翻转、字节删除、字节插入、块拼接、算术运算等多种操作，能够产生多样化的测试输入。通过这种迭代式的测试和筛选，AFL 能够逐步探索程序的各种执行路径。

AFL 的成功催生了大量改进和变体工具。AFL++ 是 AFL 的增强版本，提供了更丰富的变异策略、更好的并行支持和可扩展的插件架构。libFuzzer 是 LLVM 项目提供的库式模糊测试框架，与 AFL 的进程级模糊测试不同，libFuzzer 将模糊测试逻辑嵌入目标程序内部，以函数调用的方式执行测试，效率更高。这些工具各有特点，适用于不同的测试场景。

在嵌入式固件模糊测试领域，研究人员基于 AFL 开发了多种专用工具。这些工具通常结合固件仿真技术，在仿真环境中运行固件并注入测试数据。例如，Firmadyne、FirmAE 等工具针对路由器固件进行模糊测试；Fuzzware、P2IM 等工具针对嵌入式固件的外设建模问题，通过自动生成外设模型来支持固件在仿真环境中的运行。这些工作展示了模糊测试技术在物联网安全领域的应用潜力。
