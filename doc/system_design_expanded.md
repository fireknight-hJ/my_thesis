# 系统设计与实现（扩写版）

## 3.1 整体架构设计

### 3.1.1 设计目标

本研究提出的"基于驱动程序替换的固件仿真技术"旨在解决物联网设备固件测试中的硬件依赖问题。核心目标是在缺少真实硬件外设的情况下，使固件在通用仿真环境中稳定运行，并支持自动化测试与模糊测试。

**核心思想**：将固件中与硬件强绑定的驱动代码（寄存器读写、HAL/外设库调用、中断与 DMA 配置等）识别出来，用可在仿真环境中执行的替代实现进行等价替换，实现"固件逻辑与硬件依赖解耦"。

### 3.1.2 系统架构

系统采用分层架构设计，按"静态识别 → 智能替换 → 构建验证 → 动态反馈"的链路组织为四个核心模块：

```
┌─────────────────────────────────────────────────────┐
│              动态测试反馈闭环模块                     │
│      (仿真运行 → 日志采集 → 错误分析 → 反馈回流)      │
├─────────────────────────────────────────────────────┤
│           驱动函数分析、替换与编译模块                  │
│       (LLM 分类 → 替换策略 → 代码生成 → 编译集成)    │
├─────────────────────────────────────────────────────┤
│         驱动函数接口静态分析与识别模块                  │
│    (CodeQL 分析 → MMIO 识别 → 函数分类 → 数据模型)   │
├─────────────────────────────────────────────────────┤
│                  数据与缓存层                          │
│     (分析缓存 | 生成结果 | 编译产物 | 运行日志)        │
└─────────────────────────────────────────────────────┘
```

**图 3-1 系统整体架构图**

### 3.1.3 模块职责

**（1）静态分析与识别模块**：对固件源码进行静态分析，定位硬件相关代码，抽取驱动函数、MMIO 访问点、依赖数据结构、调用关系等信息。

**（2）分析与替换模块**：基于静态分析结果，使用大语言模型对驱动函数进行类型分类与语义保留分析，生成可替代实现，并驱动编译构建。

**（3）动态测试反馈模块**：在仿真环境中运行替换后的固件，采集运行日志、崩溃信息与覆盖率，将失败样例与错误信息结构化回流，触发重新分析与再生成。

## 3.2 驱动函数接口静态分析与识别模块

### 3.2.1 模块概述

本模块是整个方案的基础支撑层，负责自动、精准地定位固件中与硬件交互的代码，并将其抽象为标准化的"可查询知识库"。模块基于 CodeQL 静态分析引擎实现，通过预定义的查询规则从固件源码中提取驱动相关信息。

### 3.2.2 CodeQL 数据库构建

构建流程包括三个步骤：源码准备、数据库创建、索引建立。CodeQL 数据库包含抽象语法树（AST）、控制流图（CFG）、数据流图（DFG）和符号表等核心数据结构，为后续分析提供语义级支持。

### 3.2.3 MMIO 访问模式识别

MMIO（Memory-Mapped I/O）访问识别分为两步：定位 MMIO 结构体实例与捕获字段访问。

**MMIO 结构体实例定位的三种典型模式**：

| 模式 | 描述 | 示例 |
|------|------|------|
| 直接地址强转 | 将固定地址强转为外设结构体指针 | `USART_TypeDef *usart = (USART_TypeDef *)0x40011000;` |
| 间接初始化 | 通过配置表将外设基地址赋值给句柄 | `huart->Instance = USART1;` |
| 常量指针参数 | 函数以常量指针形式接收外设基址 | `void HAL_UART_Transmit(USART_TypeDef *huart, ...)` |

通过 CodeQL 查询自动识别这些模式，提取所有 MMIO 结构体实例及其字段访问点。

### 3.2.4 函数分类体系

在函数粒度上，系统区分两类硬件相关函数：

**（1）MMIO 函数**：函数内部存在直接的寄存器字段访问，如 `HAL_UART_Transmit` 直接写入 `huart->Instance->DR`。

**（2）驱动函数**：函数本身不直接访问寄存器，但通过驱动句柄间接参与硬件交互，如 `HAL_UART_Receive_IT` 设置接收回调。

根据函数语义，驱动函数划分为以下类别：

| 类别 | 缩写 | 描述 | 典型函数 |
|------|------|------|----------|
| 接收类 | RECV | 从硬件/寄存器读取数据 | `HAL_UART_Receive`, `HAL_SPI_Receive` |
| 发送类 | SEND | 将数据写入寄存器/外设 | `HAL_UART_Transmit`, `HAL_SPI_Transmit` |
| 中断类 | IRQ | 中断处理函数/回调 | `HAL_UART_IRQHandler`, `USART1_IRQHandler` |
| 初始化类 | INIT | 外设初始化 | `HAL_UART_Init`, `MX_USART1_UART_Init` |
| 控制类 | CTRL | 启停、模式切换 | `HAL_UART_Start`, `HAL_UART_Stop` |
| 轮询类 | POLL | 轮询状态位 | `HAL_UART_GetState`, `__HAL_UART_GET_FLAG` |
| DMA 配置类 | DMA | 配置 DMA 传输 | `HAL_UART_Transmit_DMA` |
| 非驱动 | NODRIVER | 不需要替换的函数 | 纯业务逻辑函数 |

### 3.2.5 数据流追踪

系统以"MMIO 字段访问"为源点，以"函数参数、局部变量、返回值"为汇点，构建数据流配置并追踪路径。追踪目的包括：

1. 确定哪些函数参数最终影响了 MMIO 访问
2. 识别需要保留的业务逻辑与需要替换的硬件操作
3. 为替换模块提供精确的语句定位信息

### 3.2.6 统一数据模型

系统定义了统一的数据模型来表示分析结果，包括 `FunctionInfo`（函数基本信息）、`MMIOAccess`（MMIO 访问点）等结构，为上层模块提供标准化查询接口。

## 3.3 驱动函数分析、替换与编译模块

### 3.3.1 智能体工作流设计

系统采用 LangGraph 框架构建图式工作流，核心包含三个逻辑节点：**推理与决策节点**、**工具调用节点**、**结果汇总节点**。

**工作流流程**：
1. 推理与决策节点接收函数源码和上下文，判断函数类别、选择替换策略
2. 如需补充信息，通过工具调用节点执行 CodeQL 查询、获取函数信息
3. 结果汇总节点将输出结构化整理并写入缓存

### 3.3.2 驱动函数自动分类

基于大语言模型实现驱动函数的自动分类。系统为 LLM 提供函数源码和分类任务提示，要求模型：

1. 读取函数源码
2. 识别 MMIO 访问（寄存器读写）
3. 判断函数的主要用途
4. 分类到预定义类别
5. 如需替换，生成替换代码

输出包含函数名、分类、是否需要替换、替换代码、置信度、分类理由等结构化信息。

### 3.3.3 替换策略体系

本研究总结出以下通用替换策略：

**策略 1：跳过硬件副作用（Skip）**

适用于：初始化类、控制类函数，寄存器写入对仿真无实际影响。

```c
// 原始代码
void HAL_UART_Enable(UART_HandleTypeDef *huart) {
    huart->Instance->CR1 |= USART_CR1_UE;
}

// 替换代码
void HAL_UART_Enable(UART_HandleTypeDef *huart) {
    (void)huart;  // 跳过硬件副作用
}
```

**策略 2：剥离硬件代码、保留控制逻辑（Strip）**

适用于：发送类函数，需要保留状态更新但移除寄存器操作。

```c
// 替换代码
HAL_StatusTypeDef HAL_UART_Transmit(UART_HandleTypeDef *huart, uint8_t *pData, uint16_t Size) {
    huart->gState = HAL_UART_STATE_BUSY_TX;
    lcmhal_uart_transmit(huart->Instance, pData, Size);  // 重定向到虚拟设备
    huart->gState = HAL_UART_STATE_READY;
    return HAL_OK;
}
```

**策略 3：替换为仿真 I/O（Redirect）**

适用于：接收/发送类函数，将操作重定向到仿真平台。

```c
// 替换代码
HAL_StatusTypeDef HAL_UART_Receive(UART_HandleTypeDef *huart, uint8_t *pData, uint16_t Size, uint32_t Timeout) {
    return lcmhal_uart_receive(huart->Instance, pData, Size, Timeout);
}
```

**策略 4：事件驱动模拟（Event）**

适用于：中断回调类函数，引入事件触发机制。

```c
// 替换代码
void HAL_UART_RxCpltCallback(UART_HandleTypeDef *huart) {
    lcmhal_event_t event = {.type = LCMHAL_EVENT_UART_RX_COMPLETE, .handle = huart};
    lcmhal_event_post(&event);
    process_received_data(huart->pRxBuffPtr);
}
```

### 3.3.4 替换验证机制

系统对生成的替换代码进行多层验证：

**（1）语法正确性检查**：使用编译器前端验证代码语法
**（2）签名匹配检查**：确保函数签名与原函数一致
**（3）返回值检查**：确保所有代码路径正确返回
**（4）替换规则检查**：验证函数名、参数、返回类型、变量声明等一致性

### 3.3.5 编译集成与错误修复

Builder Agent 负责将替换后的代码集成回工程并执行编译。编译失败时，Fixer Agent 自动解析错误信息、定位问题函数、生成修复方案并重新构建，形成编译错误修复闭环。

## 3.4 动态测试反馈闭环模块

### 3.4.1 动态执行环境设计

动态执行环境模块基于 Unicorn 模拟引擎构建，为替换后的固件提供精确、高效的模拟执行环境。模块专门针对 ARM Cortex-M 架构的嵌入式固件进行优化，支持与静态分析模块的动态信息交互。

**核心目标**包括：
- 提供与真实硬件接近的 ARM Cortex-M 微控制器执行环境
- 收集并反馈 MMIO 函数执行信息和关键执行路径
- 防止初始化代码被误判为死循环，提高模拟执行效率
- 支持模拟执行（emulate）和模糊测试（fuzz）两种模式

### 3.4.2 系统分层架构

动态执行环境采用分层架构设计，共分为 6 层：

| 层级 | 模块 | 职责 |
|------|------|------|
| 配置层 | config.py, args.py | 配置解析和管理（内存映射、符号表、函数映射） |
| 模拟层 | uc.py, fuzz.py | 指令集模拟执行 |
| 日志层 | debug.py | 日志记录和输出 |
| 模型层 | loop_queue.py, caller.py | 循环检测模型、函数调用跟踪 |
| 处理层 | handlers/common.py | HAL 函数处理 |
| 工具层 | helper/dump_config.py | 辅助工具函数 |

### 3.4.3 核心组件

**（1）配置系统**

配置系统负责解析和管理固件配置，包括内存映射、符号表、函数映射等关键信息。核心配置参数包括：Flash/RAM 基址和大小、HAL 函数映射关系、循环检测阈值等。

**（2）模拟执行引擎**

模拟执行引擎基于 Unicorn 引擎实现，提供：
- ARMv7-M Thumb 指令集支持（Cortex-M3/M4/M7）
- 精确的内存映射（Flash、RAM、外设区域独立管理）
- 通过 handler 机制模拟硬件外设行为
- VFP 浮点单元支持（兼容浮点运算固件）

**（3）循环检测系统**

嵌入式固件中存在大量循环结构，系统通过智能算法区分正常初始化循环和真正的死循环，避免误判：
- 跳过前 N 个唯一基本块（防止初始化代码误判）
- 支持白名单机制（豁免特定函数的循环检测）
- 可配置最大循环次数阈值

**（4）函数调用跟踪**

函数调用跟踪系统记录函数调用关系，特别关注 MMIO 函数的执行轨迹，为静态分析模块提供动态反馈信息。

### 3.4.4 LCMHal 专有 Handler 支持

为了精确模拟目标固件，系统实现了针对特定 HAL 函数的专门 handler。这些 handler 处理与外设的底层交互，是模拟执行的核心。

**关键 HAL 函数接口**：

| 接口名称 | 功能描述 |
|---------|---------|
| HAL_BE_ENET_ReadFrame | 从以太网接口读取数据帧 |
| HAL_BE_In | 从指定端口读取数据 |
| HAL_BE_Out | 向指定端口写入数据 |
| HAL_BE_Block_Read/Write | 块读写操作 |
| HAL_Delay, wait, wait_us | 延时函数（仿真中跳过） |

配置文件中的 `mmio_funcs` 列表定义了需要跟踪的 MMIO 函数，涵盖 ADC、DCMI、I2C、UART、GPIO 等关键外设接口。

### 3.4.5 精确内存映射管理

系统实现了精确的内存映射管理：

**（1）受保护的内存类型**：定义了 flash、ram、peripheral_ram 等不应合并的内存类型，防止不同区域被错误合并。

**（2）智能 Flash 基址检测**：自动识别代码段而非数据段作为 Flash 基址，确保固件正确加载。

**（3）VFP 浮点单元支持**：为兼容浮点运算的固件（如图像处理、网络协议栈），启用 VFP 支持。

### 3.4.6 动态反馈机制

系统通过多种方式与静态分析模块进行信息交互：

**（1）MMIO 函数执行轨迹反馈**：记录 MMIO 函数的调用地址、调用者、中断信息等。

**（2）异常循环检测反馈**：检测到异常循环时，反馈当前函数和调用者函数信息。

**（3）多通道日志输出**：提供 function.txt（函数调用记录）、debug.txt（调试信息）、lcmhal.txt（反馈专用）多个日志通道。

**（4）错误上下文反馈**：发生内存访问错误时，自动反汇编错误指令并提供详细的上下文信息。

### 3.4.7 迭代优化机制

当运行失败时，Fixer Agent 被触发进行针对性修复：

1. 分析错误类型（编译失败、链接失败、运行崩溃、运行超时）
2. 定位问题函数
3. 生成修复方案
4. 应用修复并重新构建/运行

通过"静态分析 → 智能替换 → 编译验证 → 动态测试 → 反馈优化"的完整闭环，系统不断迭代优化替换策略。

## 3.6 本章小结

本章围绕"基于驱动程序替换的固件仿真"给出了系统级设计与工程实现方法。系统采用分层模块化设计，包括：

1. **静态分析模块**：基于 CodeQL 实现驱动函数和 MMIO 访问的自动识别
2. **LLM 驱动的替换模块**：采用 LangGraph 智能体架构，实现驱动函数的自动分类和代码替换
3. **动态测试反馈闭环**：在仿真环境中运行固件，采集反馈并指导迭代优化

该设计实现了"静态识别 → 智能替换 → 构建验证 → 动态反馈"的完整闭环，为后续实验验证奠定了基础。
