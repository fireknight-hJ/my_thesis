# 3.1 系统整体架构设计

本章详细阐述基于驱动程序替换的固件仿真系统的设计与实现。3.1节介绍系统的整体架构设计，包括总体架构设计、各核心模块的职责划分以及模块间的接口设计。

## 3.1.1 系统总体架构

本系统采用分层架构设计，按"静态识别→智能替换→构建验证→动态测试反馈"的处理链路组织为四个核心模块。如图3-1所示：

```
┌─────────────────────────────────────────────────────────────────┐
│                        动态测试反馈闭环模块                        │
│         (仿真运行 → 日志采集 → 错误分析 → 反馈回流)                │
├─────────────────────────────────────────────────────────────────┤
│                    驱动函数分析、替换与编译模块                      │
│    (LLM 分类 → 替换策略选择 → 代码生成 → 编译集成)                 │
├─────────────────────────────────────────────────────────────────┤
│                  驱动函数接口静态分析与识别模块                      │
│     (CodeQL 分析 → MMIO 识别 → 函数分类 → 数据模型)               │
├─────────────────────────────────────────────────────────────────┤
│                         数据与缓存层                              │
│    (分析缓存 | 生成结果 | 编译产物 | 运行日志)                      │
└─────────────────────────────────────────────────────────────────┘
```

**图 3-1 系统整体架构图**

各模块的职责划分如下：

**（1）驱动函数接口静态分析与识别模块**  
负责对固件源码进行静态分析，定位硬件相关代码，抽取驱动函数、MMIO访问点、依赖数据结构、调用关系等信息，并以统一的数据模型对外提供接口。

**（2）驱动函数分析、替换与编译模块**  
基于上游静态分析结果，使用大语言模型对驱动函数进行类型分类与语义保留分析，生成可替代实现，同时将替代实现集成回工程并驱动构建。

**（3）动态测试反馈闭环模块**  
在仿真环境中运行替换后的固件并采集运行日志、崩溃信息与覆盖率等指标，将失败样例与错误信息结构化回流，触发重新分析与再生成。

**（4）数据与缓存层**  
贯穿上述链路，统一管理静态分析缓存、生成结果缓存、编译产物与运行日志，支持断点续跑和结果复用。

## 3.1.2 模块间接口设计

模块间通过标准化数据接口进行交互，各接口的数据结构定义如下：

**静态分析模块输出接口**：
```python
@dataclass
class FunctionInfo:
    name: str                          # 函数名
    file_path: str                     # 源文件路径
    line_range: tuple                  # (起始行, 结束行)
    signature: str                    # 函数签名
    category: str                     # 函数类别
    mmio_accesses: List[MMIOAccess]  # MMIO访问列表
    data_flow_paths: List              # 数据流路径
```

**替换编译模块输出接口**：
```python
@dataclass
class BuildArtifact:
    elf_path: str                     # 编译产物路径
    project_path: str                 # 工程路径
    modified_files: List[str]         # 修改过的文件
    success: bool                     # 构建是否成功
```

**动态测试模块反馈接口**：
```python
@dataclass
class FeedbackData:
    iteration: int                    # 迭代次数
    failure_type: str                 # 失败类型
    error_evidence: List[str]         # 错误证据
    affected_functions: List[str]     # 相关函数
    suggested_fix: str                # 修复建议
```

## 3.1.3 数据流设计

系统的核心数据流包含主数据流和反馈数据流两部分。

**主数据流**描述正常处理流程：静态分析模块接收源代码，输出函数分析结果；替换编译模块接收分析结果，生成替换代码并完成编译，输出ELF文件；动态测试模块接收ELF文件，执行仿真并输出运行结果。

**反馈数据流**描述迭代优化流程：当动态测试检测到运行失败时，将反馈数据传递给替换编译模块，触发重新分析和代码生成，形成迭代闭环，直至运行成功或达到最大迭代次数。

## 3.1.4 本节小结

本节阐述了系统的整体架构设计，包括分层模块划分、各模块职责定义、模块间接口设计以及数据流设计。系统采用四层模块化架构，各模块职责清晰、接口规范，通过主数据流和反馈数据流形成完整的处理闭环。后续章节将分别详细阐述各模块的具体设计与实现。
